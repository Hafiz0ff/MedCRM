# Database Migrations Policy

Все изменения схемы базы данных (Prisma) должны подчиняться строгим правилам для предотвращения затирания данных на продакшене и контроля дрейфа схемы (Schema Drift).

---

## Миграционная политика

1. **Одно осмысленное изменение — одна миграция**:
   Не смешивайте несколько не связанных между собой фич в одну миграцию схемы. На каждое логическое изменение структуры таблиц создавайте отдельную миграцию.

2. **Запрет db push**:
   Использование команды `prisma db push` в локальной разработке на общих базах данных или в окружениях развертывания запрещено. Всегда используйте механизм миграций `prisma migrate dev`.

3. **Именование миграций**:
   Все миграции должны следовать именованию по шаблону:
   `YYYYMMDDHHMMSS_short_kebab_scope`
   Пример: `20260601090000_add_appointment_recurrence_rule`.

4. **Дрейф Схемы (Drift Check)**:
   В CI настроен автоматический дрейф-чек с помощью команды:
   `prisma migrate diff --from-schema-datasource prisma/schema.prisma --to-schema-datamodel prisma/schema.prisma --exit-code`
   Если состояние базы данных разошлось со схемой, сборка упадет.

5. **Rollback и Безопасность**:
   Перед мержем миграции:
   - Составьте план отката (Rollback SQL Script).
   - Убедитесь, что миграция не приводит к блокировке (Exclusive Lock) больших таблиц на длительное время.
   - Избегайте деструктивных изменений (DROP COLUMN/DROP TABLE) без предварительного перехода на новую схему со сплитом фаз чтения/записи.
