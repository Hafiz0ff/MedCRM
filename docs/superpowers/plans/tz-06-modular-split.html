Разделение auth-service и Prisma-схемы на доменные сервисы

1. Назначение

README MedCRM заявляет «modular monolith with service-ready boundaries». Реально: все 11 доменов и 147 Prisma-моделей живут внутри одного приложения auth-service, schema.prisma — 3076 строк в одном файле, между модулями нет ни выделенного event bus, ни типизированных внутренних контрактов. Это не блокирует MVP, но блокирует:

  • Командное масштабирование: 2+ команды одновременно не могут спокойно мерджить large changes.
  • Performance: тяжёлый KPI-расчёт глобально замедляет registry-ответы.
  • Деплой: любое изменение требует выкатки всего auth-service.
  • K8s: один большой service не позволяет независимо масштабировать «коммуникации» (легко CPU-bound) и «EMR» (память-bound).

Цель ТЗ №6 — выполнить «правильный» Conway-разрез: вытащить из auth-service чёткие NestJS-под-приложения, вынести их Prisma-фрагменты в отдельные файлы (Prisma multi-file schemas), добавить event bus и типизированные контракты. Это не микросервисы ради микросервисов: остаёмся модульным монолитом с возможностью «отсадить» сервис в отдельный pod по мере необходимости.

Принцип «Сначала чёткие границы внутри одного процесса. Микросервис — только когда appears реальный sizing-кейс»

Преждевременный микросервис ≈ распределённый монолит, что хуже монолита по всем осям.

3. Цели и Out of Scope

Цели

  • Разрезать schema.prisma на per-domain файлы (Prisma multi-file schemas, GA в 6.x).
  • Сделать 7 доменных NestJS-приложений в backend/apps/: auth, scheduling, emr, finance, communications, inventory, analytics; плюс уже существующие api-gateway и worker (ТЗ №4).
  • Доменные модули в каждом app: integrations и operations могут жить внутри ближайшего app, отдельно — на будущее.
  • Ввести event bus (NATS JetStream, fallback Redis Streams) для inter-domain событий.
  • Типизированные внутренние контракты: shared types через backend/contracts/<domain>.contract.ts; runtime — gRPC или Nest-microservices.
  • OpenTelemetry distributed tracing сквозь все сервисы.
  • Centralized config / feature flags / service registry.
  • Keep developer-experience: localdev = docker compose up работает one-command.
  • Готовность к K8s: каждый сервис — отдельный Helm-chart (ТЗ №10).

Out of Scope

  • Полный переход к event-sourcing / CQRS по всей системе — частично (где оправдано: appointments, encounters).
  • Замена Prisma на другой ORM — не делаем.
  • Перенос на gRPC всех публичных API — только internal/internal.
  • Polyglot stack — оставляем Node/Nest везде.
  • Kubernetes deploy — отдельное ТЗ №10.
  • Изменение публичных REST-контрактов фронтенда — нет breaking changes.

4. Принципы разделения

  • Single owner per aggregate: каждая Prisma-модель принадлежит одному домену; домен — единственный, кто пишет в её таблицы.
  • Read-models — копии, не общие таблицы: если analytics нужен appointment — он получает его через event и хранит свой проекционный read-model. Никаких прямых SELECT в чужой домен.
  • Async-by-default: side effects между доменами — события. Синхронные вызовы — только когда нужна consistency или ответ.
  • Eventual consistency как явный контракт: documented latency budgets (например, «scheduling → analytics ≤ 10 сек»).
  • Backward compatibility: события и API имеют версии (v1.appointment.created); breaking — только через новый topic.
  • Один Postgres сервер, разные schemas: пока всё в одном Postgres, но логически в отдельных schema-неймспейсах (auth, scheduling, emr, ...). По мере необходимости — split DB.

6.2. Postgres schemas

В дополнение к разделению файлов выделяем Postgres schemas per domain — это даёт ещё один уровень изоляции (RLS из ТЗ №3 работает per-table, schemas даёт «доменное пространство имён»):

-- 20260801100000_postgres_schemas.sql
CREATE SCHEMA IF NOT EXISTS auth;
CREATE SCHEMA IF NOT EXISTS scheduling;
CREATE SCHEMA IF NOT EXISTS emr;
CREATE SCHEMA IF NOT EXISTS finance;
CREATE SCHEMA IF NOT EXISTS communications;
CREATE SCHEMA IF NOT EXISTS inventory;
CREATE SCHEMA IF NOT EXISTS analytics;

-- В моделях Prisma добавляем @@schema("scheduling")
-- Пример:
model Appointment {
  // ...
  @@schema("scheduling")
  @@map("appointments")
}

6.3. Совместимость с RLS (ТЗ №3)

Все политики RLS остаются на per-table уровне; tenant_id-фильтр работает одинаково. current_tenant_id() и current_user_id() — общие helpers в схеме public, доступные из любой.

6.4. Миграция данных

Переход в Postgres schemas — это rename таблиц ALTER TABLE ... SET SCHEMA .... Один аккуратный pipeline:

  1. Создать новые schemas.
  2. В одной транзакции на staging — ALTER TABLE для каждой таблицы.
  3. Прогнать integration-тесты.
  4. Применить в prod в maintenance-окно (~5 минут).

7. Сервисы и контракты

7.1. Стратегия выделения

Каждый сервис — отдельный NestJS-app в backend/apps/<name>-service с своим main.ts. Все они компилируются из одного monorepo, делят:

  • backend/core/* — общая инфраструктура (tenancy, audit, queue, communications-shared).
  • backend/contracts/* — типизированные интерфейсы для inter-service вызовов и событий.
  • backend/prisma/* — единый Prisma Client.

7.2. Контракты

Каждый сервис экспортирует:

  • REST API (через api-gateway) — публичный.
  • Internal RPC контракт (Nest-microservices с TCP или gRPC) — для inter-service синхронных вызовов.
  • Event contract — список топиков, payload-схема (Zod), versioning.
// backend/contracts/scheduling.contract.ts
export interface SchedulingService {
  createAppointment(input: CreateAppointmentInput): Promise<Appointment>;
  cancelAppointment(id: string, reason: string): Promise<void>;
  getAvailableSlots(input: GetSlotsInput): Promise<Slot[]>;
}

export const SchedulingEvents = {
  'v1.appointment.created':  z.object({ tenantId: z.string(), appointmentId: z.string(), patientId: z.string(), doctorId: z.string(), startAt: z.string(), endAt: z.string() }),
  'v1.appointment.cancelled': z.object({ tenantId: z.string(), appointmentId: z.string(), reason: z.string() }),
  'v1.appointment.completed': z.object({ tenantId: z.string(), appointmentId: z.string(), endedAt: z.string() }),
} as const;

7.4. Service discovery

  • Локально и в docker compose: hardcoded names через SERVICES_*_URL env.

8. Event bus

8.1. Выбор технологии

ОпцияProsCons
NATS JetStreamЛёгкий, persistent streams, replay, native ack.Меньше людей в команде знакомы.
Redis StreamsRedis уже в стеке.Memory-bound, ограниченная durability.
Kafka / RedpandaEnterprise-grade, лучший для high-throughput.Тяжелее в эксплуатации, избыточен на этом этапе.
RabbitMQЗнаком многим.Не event-log, мысль про replay сложнее.
Решение NATS JetStream — баланс простоты и зрелости. Redis Streams — fallback в localdev и для совсем легковесных событий.

8.2. Producer и outbox

Используем готовый outbox из ТЗ №4 (IntegrationOutbox). Расширяем для domain events:

  • Domain-операция в транзакции пишет событие в domain_outbox.
  • Outbox-relay (worker, ТЗ №4) читает unprocessed и публикует в NATS.
  • NATS гарантирует at-least-once + ordering per aggregate_id (subject suffix).

8.3. Consumer

// backend/apps/analytics-service/src/consumers/appointment.consumer.ts
import { Injectable, OnModuleInit } from '@nestjs/common';
import { NatsConsumer } from 'src/core/eventbus';

@Injectable()
export class AppointmentConsumer implements OnModuleInit {
  constructor(private readonly nats: NatsConsumer, private readonly readModel: AnalyticsReadModel) {}

  async onModuleInit() {
    await this.nats.subscribe('scheduling.v1.appointment.*', {
      durable: 'analytics-appointment-consumer',
      handle: async (msg) => {
        switch (msg.subject) {
          case 'scheduling.v1.appointment.created':   return this.readModel.onCreated(msg.data);
          case 'scheduling.v1.appointment.cancelled': return this.readModel.onCancelled(msg.data);
          case 'scheduling.v1.appointment.completed': return this.readModel.onCompleted(msg.data);
        }
      },
    });
  }
}

8.5. Schema registry

Zod-схемы событий лежат в backend/contracts/events/<domain>.ts; CI-чек ломает PR, если меняется payload без bump-версии.

8.6. Replay и debug

  • JetStream хранит события 30 дней по умолчанию (configurable).
  • NATS box CLI (nats sub) — для разработки.
  • Bull Board (из ТЗ №4) — для DLQ outbox-relay.

9. Транзакционные границы

  • В пределах одного домена: классические Prisma-транзакции (prisma.$transaction).
  • Между доменами: Saga-pattern через события + idempotency.
  • Финансовая аккуратность: создание Payment в finance + изменение Appointment.status в scheduling — два события: finance.v1.payment.captured → consumer в scheduling обновляет appointment.
  • Compensation: на случай отмены — finance.v1.payment.refunded возвращает scheduling в исходный slot (если в окне отмены).
  • Idempotency key: каждое событие имеет eventId; consumer хранит processed eventIds в собственной таблице 30 дней.
Compatibility note Не все операции выгодно превращать в saga.

На MVP-уровне 80% операций живут в одном домене и используют локальные транзакции. Saga нужны там, где есть реальный cross-domain effect.

10. Distributed observability

10.1. OpenTelemetry

  • Auto-instrumentation для NestJS (HTTP/REST), Prisma, BullMQ, NATS.
  • Context propagation: header traceparent через api-gateway → service → event → consumer.
  • Backend traces: Jaeger / Tempo / SigNoz (по выбору). Локально — Jaeger в docker compose.

10.2. Метрики

  • Prometheus exporter в каждом service.
  • RED-метрики (Rate, Errors, Duration) per endpoint.
  • Бизнес-метрики per домен: appointments_per_hour, invoices_paid, no_show_count.

10.3. Логи

  • Structured JSON, поле service, traceId, tenantId, userId.
  • Loki / CloudWatch / OpenObserve для агрегации.
  • Корреляция: один traceId связывает gateway → scheduling → analytics consumer.

10.4. Алёрты

  • Service down → P1.
  • Error rate > 1% за 5 мин → P2.
  • Event lag > 30 сек → P2.
  • DLQ size > 0 → P1.

12. Этапы работ

Спринт 1 · ~1.5 недели

  • PR refactor(prisma): multi-file schemas + postgres namespaces.
Спринт 2 · ~1 неделя

  • PR feat(core): nats jetstream eventbus + contracts.

  • Создать backend/apps/scheduling-service/ со своим main.ts и app.module.ts.

  • API gateway маршрутизирует /api/v1/appointments/* и связанное на scheduling-service.
  • Внутри auth-service оставить stubs (deprecation banner).
  • PR refactor(arch): extract scheduling-service.
Спринт 4 · ~1.5 недели

  • PR-серия refactor(arch): extract emr-service / finance-service.
Спринт 5 · ~1 неделя
