1. Назначение

В backend/package.json уже установлена bullmq, в Prisma — модели NotificationsQueue, MessageTemplate, NotificationRule, AppointmentNotification, ChatbotFlow. Но в backend/apps/auth-service/src/ нет ни одного воркера: модели наполняются seed-данными, но никто их не обрабатывает.

Цель ТЗ №4 — построить полный async-контур: producer-сервис (создаёт jobs), очереди в Redis (BullMQ), worker-приложение(я), channel-providers (Telegram Bot, SMS-агрегатор, WhatsApp Business, email), retry-policies, observability (Bull Board + Prometheus). И запустить три флагманские бизнес-фичи: appointment-reminders, missed-visit follow-up, daily KPI aggregator.

Принцип «Producer пишет в outbox + queue, worker читает и выполняет, всё идемпотентно»

Любая повторная доставка job не приводит к повторному эффекту — гарантии delivery «at-least-once».

3. Цели и Out of Scope

Цели

  • Запустить BullMQ-инфраструктуру (queues, workers, schedulers, queue-events).
  • Выделить worker-runtime как отдельное Nest-приложение в backend/apps/worker.
  • Реализовать channel-adapters: SMS, Telegram, Email; WhatsApp — заготовка.
  • Запустить три ключевых flow: appointment reminder (24h, 2h), no-show follow-up, daily KPI aggregator.
  • Templates UI (CRUD + preview + переменные {patient}, {appointment}, {clinic}).
  • Tenant-настройки каналов: connect Telegram bot, SMS gateway credentials, email SMTP.
  • Outbox-pattern для надёжной публикации событий во внешние интеграции и webhooks.
  • Bull Board UI для админов; Prometheus-метрики; OpenTelemetry-трасировка.
  • Idempotency-keys и DLQ (dead-letter queue).

Out of Scope

  • Полная маркетинг-платформа (сегментация, A/B) — это CommunicationCampaign v2, отдельное ТЗ.
  • Сложный chatbot-конструктор (drag&drop) — отдельное ТЗ.
  • Voice / IVR / SIP — отдельное ТЗ.
  • Push-уведомления native mobile — после Patient Portal MVP (ТЗ №5).
  • Полноценный workflow engine (Temporal / Camunda) — пока используем BullMQ.

4. Архитектура

Step 1 · Trigger

Producer

HTTP-эндпоинт, доменное событие (event bus), либо cron (BullMQ scheduler) кладёт job в очередь.

Step 2 · Queue

Redis (BullMQ)

Очередь хранит job с payload, retry-policy, delay, dedup-key.

Step 3 · Worker

backend/apps/worker

Pulls job, выполняет логику, использует channel-adapter, пишет результат в NotificationsQueue.

Step 4 · Provider

SMS / TG / WA / Email

Внешний сервис принимает payload; webhook возвращает delivery_status.

4.1. Топология процессов

backend/
  apps/
    api-gateway/     ← edge proxy
    auth-service/    ← REST + Domain
    worker/          ← NEW: BullMQ workers, Nest-app, не слушает HTTP
                       Запускается отдельным контейнером, scale-out independent

worker подключается к Redis и Prisma тем же способом, что и auth-service; зависимости в backend/core/* разделяются.

4.2. Гарантии

  • At-least-once delivery: BullMQ хранит job до явного completed; повторная доставка возможна при падении воркера.
  • Idempotency: каждый job имеет jobId = stableHash(payload); повторная постановка с тем же id — no-op (BullMQ removeOnComplete + jobId).
  • Retry с backoff: exponential, base 30s, до 6 попыток; затем — DLQ.
  • DLQ: отдельная очередь *.dlq; админ видит её в Bull Board и может re-queue.
  • Concurrency: каждая очередь имеет свой concurrency limit; per-tenant rate-limit через BullMQ Worker.limiter.

6. Воркеры

Каждый воркер — это NestJS-модуль с обработчиком очереди. Пример:

// backend/apps/worker/src/notifications/notifications-dispatch.worker.ts
import { Worker, Job } from 'bullmq';
import { Injectable, OnModuleInit, Logger } from '@nestjs/common';

@Injectable()
export class NotificationsDispatchWorker implements OnModuleInit {
  private readonly logger = new Logger(NotificationsDispatchWorker.name);
  private worker!: Worker<DispatchJobPayload>;

  constructor(
    private readonly channels: ChannelProviderRegistry,
    private readonly db: PrismaService,
    private readonly templates: TemplateService,
  ) {}

  onModuleInit() {
    this.worker = new Worker<DispatchJobPayload>(
      'notifications.dispatch',
      async (job) => this.handle(job),
      {
        connection: { url: process.env.REDIS_URL },
        concurrency: 20,
        limiter: { max: 60, duration: 1000 },
      },
    );

    this.worker.on('failed', (job, err) => {
      this.logger.warn(`dispatch failed id=${job?.id} attempts=${job?.attemptsMade}`, err);
    });
  }

  async handle(job: Job<DispatchJobPayload>) {
    const { tenantId, queueRecordId } = job.data;

    return await this.db.$transaction(async (tx) => {
      const rec = await tx.notificationsQueue.findFirstOrThrow({
        where: { id: queueRecordId, tenantId },
      });

      if (rec.deliveryStatus !== 'PENDING') return { skipped: rec.deliveryStatus };

      const tmpl = await this.templates.compile(rec.templateId, rec.payloadJson);

      const provider = this.channels.get(rec.channelType, tenantId);
      const result = await provider.send({
        to: tmpl.recipient,
        subject: tmpl.subject,
        body: tmpl.body,
        meta: { queueId: rec.id, appointmentId: rec.appointmentId },
      });

      await tx.notificationsQueue.update({
        where: { id: rec.id },
        data: {
          deliveryStatus: result.status,
          retryCount: { increment: 1 },
          processedAt: new Date(),
        },
      });

      return result;
    });
  }
}

6.1. Идемпотентность и dedup

await queue.add('dispatch', payload, {
  jobId: `nq:${queueRecordId}`,    // unique key
  attempts: 6,
  backoff: { type: 'exponential', delay: 30_000 },
  removeOnComplete: { count: 1000, age: 60 * 60 * 24 * 7 },
  removeOnFail:     { count: 5000, age: 60 * 60 * 24 * 30 },
});

6.2. Schedule scanner

// каждые 5 минут
async scan() {
  const now = new Date();
  const horizon24 = new Date(now.getTime() + 24 * 3600_000);
  const horizon2  = new Date(now.getTime() + 2  * 3600_000);

  for (const tenant of await this.tenants.activeForReminders()) {
    const appts = await this.prisma.appointment.findMany({
      where: {
        tenantId: tenant.id,
        startAt: { gte: now, lte: horizon24 },
        status:  { in: ['SCHEDULED', 'CONFIRMED'] },
        notifications: { none: { type: 'REMINDER_24H' } },
      },
      include: { patient: true, doctor: true },
    });

    for (const a of appts) {
      await this.enqueueReminder(a, 'REMINDER_24H');
      if (a.startAt < horizon2) await this.enqueueReminder(a, 'REMINDER_2H');
    }
  }
}

7. Notification flows

7.1. Appointment reminder

Окно Канал-приоритет Текст (пример)
−24h Telegram → WhatsApp → SMS {patient.firstName}, напоминаем: завтра в {appointment.startAt|time} вас ждёт {doctor.lastName}. Подтвердить: /yes Отменить: /no
−2h SMS → Telegram Через 2 часа: {clinic.name}, кабинет {appointment.room.code}. Адрес: {clinic.addressShort}.
+30мин (не пришёл) Telegram → SMS Мы не дождались Вас. Хотите перенести? /reschedule
День после Telegram → email Спасибо за визит. Оцените: /1 /2 /3 /4 /5

7.2. No-show follow-up

  1. Через 30 минут после endAt со статусом NO_SHOW или SCHEDULED (не check-in): автоматически создаётся conversation с пациентом, отправляется reminder.
  2. Если пациент отвечает /reschedule — bot предлагает 3 ближайших слота у того же врача.
  3. Если ответа нет 24 часа — задача закрывается; pattern-репорт «постоянные no-show» подсвечивается в патиент-карте.

7.3. Daily KPI aggregator

  • Cron 01:00 локального tenant-timezone.
  • Считает: appointments_total, completed, no_show, average ticket, gross revenue, debt aging, доктор-load.
  • Записывает в FinancialDailyAggregate, DoctorKpiMetric, DwFactAppointment, DwFactPayment.
  • Опционально: отправляет краткую сводку в Telegram-чат тенанта (главврач, директор).

7.4. Operational hourly

  • Каждый час: пересчёт текущей загрузки кабинетов, врачей, длины очереди, среднего времени ожидания.
  • Публикуется в Socket.IO канал operations:<tenantId> — обновляет dashboard live.

8. Templates studio

UI для редактирования шаблонов в Settings → Коммуникации → Шаблоны.

8.1. Возможности

  • CRUD MessageTemplate с разделением по channelType (TELEGRAM / WHATSAPP / SMS / EMAIL) и languageCode.
  • Системные шаблоны (isSystem: true) не редактируются полностью — только override на уровне тенанта.
  • Переменные через Handlebars-like синтаксис: {{patient.firstName}}, {{appointment.startAt | date "dd.MM HH:mm"}}, {{clinic.name}}.
  • Preview справа: вставляются sample-данные.
  • Длина SMS-сообщения подсчитывается (160/70 chars в зависимости от кодировки).
  • Локализация ru / tj / en / uz; fallback на ru.
  • Линтер шаблона: проверка переменных против whitelist, проверка незакрытых выражений.

8.2. Рендеринг

Handlebars.registerHelper('date', (val, fmt) => format(val, fmt));

8.3. Notification rules

Tenant-admin настраивает в Settings → Правила:

  • Включить/выключить REMINDER_24H, REMINDER_2H, NO_SHOW_FOLLOWUP, POST_VISIT_NPS.
  • Изменить окно (например, REMINDER_24H → 36 часов).
  • Изменить порядок каналов (Telegram → SMS, или наоборот).
  • Quiet hours (не отправлять с 22:00 до 08:00).

9.3. Telegram-bot per tenant

  • В Settings → Коммуникации → Telegram tenant-admin вводит botToken; backend регистрирует webhook на POST /webhooks/telegram/<tenantHash>.
  • Bot хранит конверсации, отвечает на /yes, /no, /reschedule.
  • Patient ⇄ TG chat связывается через one-time deeplink при первом визите.
  • Текст сообщения берётся из шаблонов; кнопки (inline keyboard) — из конфигурации flow.

9.4. WhatsApp opt-in

  • Шаблоны должны быть pre-approved Meta (HSM template).
  • 24-часовое окно для free-form ответа.
  • Запрет рекламных рассылок без явного opt-in (CommunicationPreference).

9.5. SMS

  • Маршрутизация: per-tenant выбор gateway; primary + fallback.
  • Sender ID per tenant (одобренный): обычно имя клиники.
  • Учёт стоимости в CommunicationMetric.

9.6. Email

  • SPF/DKIM/DMARC настроить per-domain тенанта.
  • Шаблоны через MJML → HTML; рендерим в воркере, не на лету.
  • Опционально tracking pixel (с opt-in).

10. Outbox для интеграций

Чтобы не терять события при отказе внешнего сервиса (Telegram, лаборатория, ОМС, фискальная касса):

  1. Domain-операция пишет событие в integration_outbox в той же транзакции, что и доменные данные.
  2. Outbox-relay (cron каждые 5 секунд) читает unprocessed events и кладёт в queue integrations.outbox.
  3. Worker отправляет в провайдер (HTTP webhook, HL7 endpoint, FHIR push); при успехе — помечает запись processed_at.
  4. При фейле — exponential backoff до 24 часов, потом DLQ.

10.1. Новая модель

model IntegrationOutbox {
  id            String   @id @default(uuid()) @db.Uuid
  tenantId      String   @map("tenant_id") @db.Uuid
  eventType     String   @map("event_type") @db.VarChar(120)
  aggregateType String   @map("aggregate_type") @db.VarChar(120)
  aggregateId   String   @map("aggregate_id") @db.Uuid
  payloadJson   Json     @map("payload_json")
  scheduledAt   DateTime @default(now()) @map("scheduled_at") @db.Timestamptz(6)
  attemptCount  Int      @default(0) @map("attempt_count")
  lastError     String?  @map("last_error")
  processedAt   DateTime? @map("processed_at") @db.Timestamptz(6)
  state         String   @default("PENDING") @db.VarChar(20)   // PENDING | IN_PROGRESS | DONE | FAILED

  @@index([state, scheduledAt])
  @@index([tenantId, aggregateType, aggregateId])
  @@map("integration_outbox")
}

11. Observability воркеров

  • Bull Board UI (@bull-board/express + @bull-board/api) на пути /internal/v1/queues; доступ — только Platform Admin (RBAC).
  • Prometheus метрики: bullmq_queue_jobs_total{queue, state}, bullmq_queue_jobs_duration_seconds, bullmq_queue_jobs_failed_total.
  • OpenTelemetry трасировка: producer создаёт span, в payload передаётся traceparent; worker продолжает span. Sentry для ошибок.
  • Logs: structured JSON, поле jobId, queue, tenantId; Loki / CloudWatch.
  • Alerts: DLQ size > 0 → P1; lag > 5 минут на notifications.dispatch → P2; провайдер-success-rate < 95% → P3.
  • Bull Board access: за reverse-proxy с RBAC; запрещён публично.

14. Acceptance criteria

  • Worker-приложение запускается отдельным контейнером и подключается к Postgres+Redis.
  • Все ключевые очереди созданы: notifications.dispatch, schedule-scan, no-show, kpi.daily/hourly, outbox, audit.checkpoint, maintenance.cleanup, documents.render.
  • Idempotency: повторная постановка job с тем же payload → no-op.
  • Reminder 24h/2h отправляется минимум по одному реальному каналу (SMS либо Telegram); пациент получает сообщение.
  • Chatbot обрабатывает /yes, /no, /reschedule; статус appointment меняется соответствующе.
  • No-show follow-up создаёт conversation и сообщение через 30 минут после endAt.
  • KPI daily/hourly worker наполняет агрегаты; ошибок не приводит к битым данным (idempotent upsert).
  • Outbox-relay доставляет события на webhook subscriber; при отказе экспоненциальный retry до 24 часов.
  • Bull Board доступен под Platform Admin; показывает счётчики, jobs, retries, DLQ.
  • Prometheus-метрики экспортируются; Grafana-дашборд с базовыми панелями добавлен в репо.
  • Sentry получает ошибки воркеров с контекстом jobId + tenantId.
  • Templates UI: CRUD + preview + sms length counter + lint работают.
  • Notification rules UI: вкл/выкл REMINDER_24H/REMINDER_2H/NO_SHOW/POST_VISIT + quiet hours.
  • E2E-тест: создать appointment → пройти полный цикл (reminder → no-show → followup → reschedule) — green.
  • DLQ имеет re-queue button в Bull Board.

15.2. Риски

Риск Митигация
Дубль-отправка reminder из-за повторных jobs. jobId = stableHash(appointmentId+type); уникальные (notificationsQueue.appointmentId, type).
SMS-spam при баге scanner. Rate-limit per-tenant в Worker (max 60 dispatch/sec); circuit-breaker per channel; ручной kill-switch.
Telegram Bot blocked пациентом → reminder не доходит. Fallback на SMS из channel-priority; запоминаем block-статус, не пытаемся отправлять снова.
WhatsApp HSM-шаблон не одобрен Meta. WhatsApp как «next-step»; продвигаем Telegram + SMS первым.
Worker падает на одной job → блокирует очередь. Concurrency > 1, isolate per job; sentry capture; auto-restart pod через k8s.
Outbox-таблица растёт безгранично. Retention 30 дней с processed_at; cleanup worker.
Bull Board exposed публично. Mounted под /internal/v1 с RBAC, gateway отрезает извне.
Сложно отлаживать distributed-job. OpenTelemetry traceId сквозной; jobId в каждом лог-записи; replay из Bull Board.
Quiet hours различаются по tenant timezone. Хранить Tenant.timezone; scanner работает в локальном таймзоне tenant.
Регуляторный риск рассылок (СПАМ). Opt-in через CommunicationPreference; reminder = транзакционный (не маркетинговый); кампании — отдельно с явным opt-in.
