MedCRM · ТЗ №7 · EMR / Clinical Excellence

Электронная медицинская карта с клиническим интеллектом

Превратить EMR из набора текстовых полей в полноценную клиническую систему: структурированные протоколы приёма, справочники ICD-10/ICD-11/MЭС/МНН, проверка взаимодействий и аллергий, специализированные форматы (одонтограмма, протоколы УЗИ, гинекология, педиатрия), импорт лабораторных результатов с пометками отклонений, витал-параметры с алёртами, шаблоны рецептов и e-подпись врача, телемедицина-готовность.

1. Назначение

В текущем MedCRM существуют модели Encounter, EncounterDiagnosis, EncounterDocument, MedicalRecord, Prescription, ServiceOrder. Они хранят произвольный текст. Это работает на «терапевт-консультанта», но не подходит для:

  • Узких специалистов (стоматолог, УЗИ, гинеколог, педиатр) с жёсткими шаблонами протокола.
  • Лекарственной безопасности (нет проверок drug-drug interactions и аллергий).
  • Аналитики (нельзя посчитать «сколько случаев ОРВИ за квартал»).
  • Обмена данных (FHIR-export требует структуру, а не свободный текст).
  • Электронной подписи документов и отчётности в надзорные органы.

Цель ТЗ №7 — добавить клинический интеллект, не превращая интерфейс врача в «бюрократический ад». Структурировать там, где это даёт ценность (диагнозы, назначения, vitals, лаборатория, аллергии); оставить свободный текст там, где он органичен (anamnesis, status localis, план лечения).

Принцип «Структура для безопасности и аналитики, свободный текст для рассуждения»

Каждое поле — либо явно структурировано (ICD-код, dose), либо явно «нарратив» (заметка врача). Никаких «полу-структурированных» костылей.

3. Цели и Out of Scope

Цели

  • Шаблоны протоколов с structured-fields + narrative-fields (chief complaint, anamnesis, status, plan).
  • Шаблоны per-специальность: терапевт, педиатр, гинеколог, стоматолог, УЗИ, дерматолог, оториноларинголог.
  • Справочники: ICD-10, ICD-11 (готовность), МКБ-10 RU/TJ, МНН, торговые наименования, единицы измерения, формы выпуска, ATX-классификация, аллергены.
  • Allergy check: при назначении препарата проверяется список известных аллергий пациента.
  • Drug-drug interactions: предупреждение при назначении препаратов с известными взаимодействиями.
  • Прескрипция через структуру (МНН + дозировка + кратность + длительность + путь введения).
  • E-подпись encounter и prescription (через UserCredential WebAuthn из ТЗ №3).
  • Витал-параметры с алёртами (например, давление > 180/110 → красный).
  • Лабораторные результаты с пометками normal/abnormal/critical (после ТЗ №8 LIS).
  • Одонтограмма (карта зубов) с историей вмешательств.
  • УЗИ-протоколы со встроенным DICOM-viewer-link (DICOM хранение — в ТЗ №8).
  • Audit + неизменяемость подписанного encounter; «отменить подпись» возможно только через формальный процесс.
  • FHIR R5-маппинг для каждой структурированной сущности.

Out of Scope

  • Голосовой ввод и AI-резюмирование приёма — отдельное ТЗ.
  • NLP-извлечение из исторического свободного текста — отдельная задача.
  • Полная база лекарств (RxNorm / SNOMED CT) — лицензируется отдельно; начинаем с открытых источников + ATX.
  • Полная HL7/FHIR cross-system sync — ТЗ №8.
  • DICOM хранилище — ТЗ №8.
  • Reportable conditions / эпиднадзор — отдельное специализированное ТЗ.
  • Решения о терапии «AI рекомендует» — не вторгаемся в клинический процесс без отдельной валидации.

4. Ключевые понятия

  • Encounter — один эпизод медицинской помощи (визит). Имеет статус: OPEN → SIGNED → AMENDED.
  • ClinicalTemplate — описание формы протокола для специальности и типа визита.
  • EncounterSection — секция в encounter (соответствует секции шаблона).
  • Diagnosis — структурированный диагноз: код ICD + ICD-version + тип (primary/secondary), статус (active/resolved).
  • Allergy — известная аллергия пациента: agent (лекарство / еда / другое), severity, дата подтверждения.
  • VitalSign — измерение: тип (BP/HR/Temp/SpO2/...), значение, единица, момент, контекст.
  • Prescription — рецепт: одна или несколько PrescriptionItem со структурой дозы.
  • ServiceOrder — направление на исследование / процедуру.
  • LabReport — результат лабораторного исследования: набор LabResultItem с reference-range.

6. Справочники

6.1. ICD-10 / ICD-11 / МКБ-10 RU

  • Хранение в ReferenceIcdCode: {code, version, parentCode, title, titleRu, titleTj, titleEn, isLeaf, isActive}.
  • Источники: ICD-10 WHO (free), МКБ-10 Минздрав РФ, ICD-11 WHO API (subscription tier — на будущее).
  • Импорт через seed-скрипт; обновление раз в полгода через CI.
  • API /api/v1/dicts/icd?q=...&version=10 с автокомплитом < 100ms.

6.2. МНН и торговые наименования

  • ReferenceInn: МНН (международное непатентованное наименование), ATX-код, формы выпуска.
  • ReferenceMedicinalProduct: торговое название, производитель, дозировка, форма, страна.
  • Источники: открытые реестры (ГРЛС РФ, реестры РТ/УЗ, EMA для интернационального уровня); опционально RxNorm под лицензию.
  • Семантика: один МНН → много торговых наименований; назначение через МНН (фарм-стандарт).

6.3. Аллергены

  • ReferenceAllergen: code (RxNorm / WHO-IPCS), category (drug/food/environment/other), title.
  • Drug-аллерген имеет связь с МНН и ATX.

6.4. Drug-drug interactions

  • ReferenceDdi: пара МНН (или ATX-class) + severity (contraindicated/major/moderate/minor) + механизм + рекомендация.
  • Источник: открытые DDI базы (NCBI, Lexicomp lite). В будущем — лицензионная.
  • API /api/v1/cds/ddi-check принимает список МНН и возвращает список взаимодействий.

6.5. Единицы и формы выпуска

  • ReferenceUnit: mg, g, ml, IU, drop, tab, ... с UCUM-кодами.
  • ReferenceDosageForm: tab, cap, syrup, drops, injection, ...
  • ReferenceRoute: PO, IV, IM, SC, TOP, INH, ...

7. Clinical Decision Support (CDS)

7.1. Правила

  1. Allergy alert: при добавлении PrescriptionItem проверяется match с PatientAllergy. Severity → блок (contraindicated), warning (major) или info (minor).
  2. Drug-drug interaction: pairwise check между всеми active-prescriptions пациента + текущим назначением.
  3. Dose check: рекомендуемая доза по возрасту, весу (для педиатрии). Если выходит за диапазон — warning.
  4. Duplicate therapy: уже есть active prescription того же МНН.
  5. Renal/hepatic: если в карте есть метка «почечная недостаточность» / «печёночная» и препарат требует адаптации дозы — alert.
  6. Pregnancy: если пациентка беременна (маркер в карте) — категория FDA-pregnancy (A/B/C/D/X) проверяется.
  7. Critical lab: если последний lab-результат содержит «critical» — alert на encounter.

7.2. UI alert

  • Banner вверху encounter с цветом severity.
  • Tooltip с источником данных и ссылкой на DDI-запись.
  • Override: врач может явно «override» с обязательным комментарием. Override логируется в audit (ТЗ №3).
  • Метрики per tenant: сколько override, по каким DDI — admin-dashboard.

7.3. Архитектура

CDS — отдельный модуль в emr-service (ТЗ №6). Stateless engine с правилами; принимает {patient context, prescriptions, vitals}, возвращает alerts[]. Может быть вынесен в отдельный сервис в будущем.

8. Рецепты и e-подпись

8.1. Структура

8.2. E-подпись

  • Encounter status: OPEN → SIGNED.
  • Подпись — через WebAuthn (ТЗ №3): врач touch ID / YubiKey подтверждает.
  • Хеш контента encounter подписывается; результат сохраняется в EncounterSignature.
  • Подписанный encounter становится immutable. Изменения возможны через AMENDED-flow: создаётся версия с reason, original остаётся.
  • Prescription наследует подпись encounter.

8.3. PDF рецепта

  • Шаблон PDF (формат локальный: РФ № 107-1/у, РТ согласно МЗ РТ).
  • QR-код с подписанной payload (для проверки в аптеке).
  • Подпись и печать врача → визуальное представление подписи + криптографическая подпись валидируется.
  • Локализация ru/tj/uz.
  • Рендеринг — worker (ТЗ №4).

9. Назначения и направления (Service Orders)

  • ServiceOrder: lab / radiology / consultation / procedure.
  • Структура: код услуги (из Service), приоритет (routine/urgent/stat), клинические показания (диагнозы), ожидаемая дата.
  • Lab order имеет нек-то LabTestPanel (например, «ОАК», «биохим. профиль»).
  • Radiology order связан с DICOM modality (CT / MRI / US / X-Ray).
  • Направления печатаются как PDF; QR-код для приёма в лаборатории.
  • Статус: ORDERED → IN_PROGRESS → COMPLETED → REPORTED.
  • Связь с LabReport / DicomStudy после выполнения.

  • Алёрты: critical results → уведомление врачу (через ТЗ №4).
  • FHIR Observation ресурс — natural mapping.

11. Витал-параметры

model VitalSign {
  id            String   @id @default(uuid()) @db.Uuid
  tenantId      String   @map("tenant_id") @db.Uuid

}
  • Алёрт-таблица per type: BP_SYS > 180 → CRITICAL; SPO2 < 90 → CRITICAL; ...
  • UI отображает значение со знаком «↑/↓» и цветом.
  • Графики тренда за период (Recharts из ТЗ №2).
  • FHIR Observation.

12. Стоматология: одонтограмма

  • Графическая карта зубов (FDI или Universal система), интерактивная.
  • DentalChartEntry: {patientId, toothCode (FDI 11..48 или mol 1..32), surface (M/D/B/L/O), procedure, status, date, encounterId}.
  • UI: SVG-схема, hover показывает историю манипуляций; click → форма ввода (диагноз/процедура).
  • Печать «история стоматологического лечения» с одонтограммой как PDF.
  • Шаблоны процедур: лечение кариеса, эндодонтия, удаление, протезирование, имплантация.
  • Расширение: имплант-passport (производитель, lot, дата).

13. УЗИ и визуализация

  • Шаблоны протоколов УЗИ: брюшная полость, малый таз, щитовидная, молочные железы, акушерство (срок беременности).
  • Структурированные поля: размеры органов в мм, эхогенность, контуры (выбор из списка).
  • Прикрепление изображений (JPEG/PNG) к протоколу; через MinIO с подписанными URL.
  • Ссылка на DICOM Study в PACS (ТЗ №8) — через studyInstanceUid и viewer-URL.
  • Шаблоны для радиолога: рентген, КТ (по системам), МРТ (по областям).

14. Telehealth-готовность

Без интеграции видео-канала, но с подготовкой:

  • Appointment.kind: IN_PERSON | TELEMEDICINE.

18. Acceptance criteria

  • Справочники ICD-10, МНН, торговых наименований, DDI, аллергенов, единиц — импортированы; автокомплит работает < 100ms.
  • ≥ 10 системных шаблонов протоколов поставляются «из коробки», tenant может override.
  • Encounter имеет структурированные секции; диагнозы выбираются из ICD.
  • Vitals хранятся структурированно, отображаются с алёртами и трендом.
  • Аллергии и хронические состояния хранятся в карточке пациента.
  • Назначения — структурированные (МНН + dose + route + frequency + duration).
  • CDS работает: allergy, DDI, dose, duplicate, pregnancy. Override требует комментарий и пишется в audit.
  • Encounter подписывается через WebAuthn; signed = immutable.
  • Amendments возможны, но видны как отдельная версия с историей.
  • PDF выписки и рецепта — соответствуют локальным формам; QR-код валиден.
  • Одонтограмма работает; история процедур по зубам отображается.
  • УЗИ-протоколы с прикреплением изображений.
  • Лабораторные результаты импортируются и отображаются с reference-range.
  • Critical lab триггерит уведомление врачу.
  • FHIR-export для encounter/condition/medication/observation работает.
  • Audit-log включает каждое подписание, override, amendment.
