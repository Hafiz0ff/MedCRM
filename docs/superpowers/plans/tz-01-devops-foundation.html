1. Назначение

MedCRM позиционируется как production-oriented SaaS МИС. Однако в репозитории отсутствует базовый инженерный контур: нет CI, нет линтеров, нет pre-commit, нет фронтенд-тестов и нет автоматического обновления зависимостей. Это означает, что любой PR в текущем состоянии может вмержить тайп-эррор, сломать миграции, поломать e2e или добавить уязвимую зависимость — и команда узнает об этом только на проде.

Цель ТЗ №1 — установить «инженерный пол», ниже которого PR не должен попасть в main. Все остальные ТЗ (дизайн-система, безопасность, очереди, портал пациента) строятся поверх этого фундамента.

Принцип «Локально проходит — в CI проходит — в проде стабильно»

Каждая проверка, которая ловит ошибку в CI, должна быть запускаема локально одним npm-скриптом и автоматически в pre-commit/pre-push.

3. Цели и Out of Scope

Цели

  • CI на GitHub Actions, обязательный перед merge в main.
  • Single source of truth для линтера и формата (один конфиг ESLint и Prettier на репо).
  • Husky + lint-staged: pre-commit прогоняет lint только на staged-файлах, commit-msg валидирует Conventional Commits.
  • Prisma format + validate + drift-check в CI.
  • Docker build обоих образов (gateway, auth, frontend) в CI как smoke-проверка.
  • Renovate-бот обновляет зависимости еженедельно отдельными PR.
  • Security gates: npm audit, secret-scan, OWASP-baseline.
  • Бейджи статуса CI в README.

Out of Scope

  • Полноценный CD в Kubernetes/Helm — это отдельное ТЗ (enterprise-зрелость).
  • Observability stack (Sentry, OTel, Prometheus, Loki) — отдельное ТЗ.
  • Load testing (k6) — отдельное ТЗ.
  • Полные contract-тесты gateway → auth-service — частично в Acceptance, полноценно — после разделения сервисов.
  • Замена существующих CI/CD инструментов на GitLab CI или Circle.

5. CI pipeline

Pipeline разбит на несколько workflow, чтобы можно было запускать независимо:

ci.yml

Триггер: pull_request + push в main.

  • install (npm ci, cache .npm и .next/cache)
  • lint (root, backend, frontend)
  • format-check (prettier --check)
  • typecheck (backend + frontend)
  • prisma validate + prisma format check + migrate diff dry-run
  • backend tests: gateway specs + e2e suites
  • frontend tests: vitest + playwright
  • docker build smoke (api-gateway, auth-service, frontend)

codeql.yml

SAST на TypeScript. Триггер: push в main, weekly cron.

  • codeql-action/init
  • codeql-action/analyze
  • Результаты → Security tab GitHub.

release-please.yml

Автоматический changelog и release-PR.

  • Слушает main
  • Парсит Conventional Commits
  • Поддерживает release-please-config.json с workspaces
  • Создаёт release PR; merge → git tag + GitHub Release.

5.1. Скелет ci.yml

name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

concurrency:
  group: ci-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  install:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
          cache: npm
      - run: npm ci
      - uses: actions/cache@v4
        with:
          path: |
            node_modules
            backend/node_modules
            frontend/node_modules
          key: deps-${{ hashFiles('**/package-lock.json') }}

  lint-and-format:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 24, cache: npm }
      - run: npm ci
      - run: npm run lint
      - run: npm run format:check

  typecheck:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 24, cache: npm }
      - run: npm ci
      - run: npm --workspace backend run typecheck
      - run: npm --workspace frontend run typecheck

  prisma:
    needs: install
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:17-alpine
        env:
          POSTGRES_USER: medcrm
          POSTGRES_PASSWORD: medcrm_password
          POSTGRES_DB: medcrm
        ports: ["5432:5432"]
        options: >-
          --health-cmd "pg_isready -U medcrm"
          --health-interval 10s --health-timeout 5s --health-retries 5
    env:
      DATABASE_URL: postgresql://medcrm:medcrm_password@localhost:5432/medcrm?schema=public
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 24, cache: npm }
      - run: npm ci
      - run: npx --workspace backend prisma format --check
      - run: npx --workspace backend prisma validate
      - run: npm --workspace backend run prisma:migrate
      - name: Drift check
        run: npx --workspace backend prisma migrate diff \
          --from-schema-datasource backend/prisma/schema.prisma \
          --to-schema-datamodel    backend/prisma/schema.prisma \
          --exit-code

  backend-tests:
    needs: [install, prisma]
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:17-alpine
        env: { POSTGRES_USER: medcrm, POSTGRES_PASSWORD: medcrm_password, POSTGRES_DB: medcrm }
        ports: ["5432:5432"]
      redis:
        image: redis:7-alpine
        ports: ["6379:6379"]
    env:
      DATABASE_URL: postgresql://medcrm:medcrm_password@localhost:5432/medcrm?schema=public
      REDIS_URL: redis://localhost:6379
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 24, cache: npm }
      - run: npm ci
      - run: npm --workspace backend run prisma:generate
      - run: npm --workspace backend run prisma:migrate
      - run: npm --workspace backend run prisma:seed
      - run: npm --workspace backend run test:gateway
      - run: npm --workspace backend run test:e2e

  frontend-tests:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 24, cache: npm }
      - run: npm ci
      - run: npm --workspace frontend run test
      - run: npx --workspace frontend playwright install --with-deps chromium
      - run: npm --workspace frontend run test:e2e

  docker-build:
    needs: install
    runs-on: ubuntu-latest
    strategy:
      matrix:
        target: [api-gateway, auth-service, frontend]
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - name: Build ${{ matrix.target }}
        uses: docker/build-push-action@v5
        with:
          context: ${{ matrix.target == 'frontend' && './frontend' || './backend' }}
          file: ${{ matrix.target == 'frontend' && './frontend/Dockerfile' || './backend/infrastructure/docker/Dockerfile' }}
          build-args: APP_NAME=${{ matrix.target }}
          push: false
          tags: medcrm/${{ matrix.target }}:ci
          cache-from: type=gha
          cache-to: type=gha,mode=max

  gitleaks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
Branch protection Включить required checks в GitHub Settings → Branches → main:
  • lint-and-format, typecheck, prisma, backend-tests, frontend-tests, docker-build, gitleaks
  • Require linear history, require signed commits (опционально), require review from CODEOWNERS.

6. Lint / Format / Hooks

6.1. ESLint flat config

Один root-конфиг eslint.config.mjs + расширения в каждом workspace. Включить:

  • @eslint/js recommended;
  • typescript-eslint strict-type-checked;
  • eslint-plugin-import с правилами на import/order, import/no-cycle;
  • eslint-plugin-unicorn recommended (filename-case, prefer-node-protocol);
  • eslint-plugin-promise;
  • в frontend — @next/eslint-plugin-next core-web-vitals, eslint-plugin-jsx-a11y, eslint-plugin-react, eslint-plugin-react-hooks.

6.2. Prettier

Конфиг .prettierrc.json в корне:

{
  "semi": true,
  "singleQuote": true,
  "trailingComma": "all",
  "printWidth": 100,
  "tabWidth": 2,
  "plugins": ["prettier-plugin-prisma"]
}

.prettierignore включает dist/, .next/, node_modules/, package-lock.json, генерёшку backend/prisma/generated.

6.3. Husky + lint-staged

// package.json (root)
{
  "scripts": {
    "prepare": "husky",
    "lint": "eslint . --max-warnings=0",
    "format": "prettier --write .",
    "format:check": "prettier --check ."
  },
  "lint-staged": {
    "*.{ts,tsx,js,jsx}": ["eslint --fix --max-warnings=0", "prettier --write"],
    "*.{json,md,yml,yaml,css}": ["prettier --write"],
    "backend/prisma/schema.prisma": ["prisma format"]
  }
}

.husky/pre-commit:

npx lint-staged
npx gitleaks protect --staged --redact --no-banner

.husky/commit-msg:

npx --no -- commitlint --edit "$1"

.husky/pre-push (опционально, быстрая защита перед push):

npm --workspace backend run typecheck &
npm --workspace frontend run typecheck &
wait
Compatibility Husky v9 не использует .husky/_/husky.sh — миграция тривиальная, см. https://typicode.github.io/husky/migrating-from-v4.html.

7. Commits и PR-дисциплина

7.1. Conventional Commits

Принимаем формат type(scope): subject. Допустимые type:

feat

новая фича для пользователя

fix

баг-фикс

refactor

переработка без изменения поведения

perf

улучшение производительности

docs

документация

test

тесты

chore

служебные изменения, deps

ci

конвейер сборки

scope — модуль из текущей структуры: auth, emr, finance, reception, scheduling, gateway, frontend, prisma, infra.

7.2. PR-шаблон

Файл .github/pull_request_template.md должен содержать:

  • Что и зачем (бизнес-описание);
  • Технические детали (как реализовано);
  • Скриншоты / Loom для UI-изменений;
  • Migration-плэн (если затрагивает Prisma);
  • Чек-лист: typecheck, lint, tests, миграции (`prisma migrate dev`), Storybook (после ТЗ №2).

7.3. CODEOWNERS

Назначить ответственных по доменам — улучшает качество ревью и распределяет ответственность. 

/backend/apps/auth-service/src/auth/        @core-team
/backend/apps/auth-service/src/emr/         @medical-team
/backend/apps/auth-service/src/finance/     @finance-team
/backend/prisma/                            @data-team
/frontend/modules/                          @frontend-team
/frontend/shared/ui/                        @design-system-team
/.github/                                   @platform-team

8. Prisma и миграции

147 моделей в одном файле — это технический долг. До разделения на доменные файлы (ТЗ для отдельного этапа) ставим минимально обязательные правила:

  • В CI всегда prisma format --check и prisma validate.
  • Migration policy: одно осмысленное изменение — одна миграция. Запрещено схлопывать несколько фич в один файл.
  • Имена миграций: YYYYMMDDHHMMSS_short_kebab_scope (например 20260601090000_add_appointment_recurrence_rule).
  • Drift detection в CI: prisma migrate diff — если детектируется drift между schema.prisma и migrations, билд падает.
  • Запретить prisma db push в репозитории — только prisma migrate.
  • В backend/prisma/README.md описать процесс ревью миграций (rollback-план, ETA, влияние на индексы, lock-time на больших таблицах).
Будущее Подготовить переход на prismaSchemaFolder (preview-feature).

Это позволит разбить schema.prisma на schema/auth.prisma, schema/emr.prisma, schema/finance.prisma и т.д. Включается отдельным мини-ТЗ, но планируется уже сейчас.

9. Управление зависимостями

9.1. Renovate

Файл renovate.json в корне. Базовый профиль:

  ],
  "schedule": ["before 6am on monday"],

      "matchDepTypes": ["devDependencies"],

    {
      "matchPackagePatterns": ["^@nestjs/"],
      "groupName": "nestjs"
    },
    {
      "matchPackagePatterns": ["^@prisma/", "^prisma$"],
      "groupName": "prisma"
    },
    {

      "groupName": "frontend core"
    },
    {
      "matchUpdateTypes": ["major"],
      "dependencyDashboardApproval": true
    }

    "enabled": true,
    "labels": ["security"]
  }
}

9.2. npm audit + Snyk

  • В CI: npm audit --audit-level=high — fail при high/critical;
  • Опционально snyk test --severity-threshold=high при наличии токена;
  • Уязвимости с CVSS ≥ 7.0 блокируют merge.

9.3. Node version pinning

Создать файл .nvmrc с 24, добавить "engines": { "node": ">=24" } в корневой package.json. В Dockerfile зафиксировать FROM node:24-alpine.

10. Security gates

Gate Инструмент Когда срабатывает Действие
Secret scan Gitleaks pre-commit + CI Блок коммита и PR при детекте секрета.
SAST GitHub CodeQL (TS) PR + weekly cron Уязвимости -> Security tab; high → PR fail.
SCA npm audit + Renovate vuln alerts CI на каждый PR Блок при high/critical.
Docker image scan Trivy (опционально) после docker-build job Отчёт; high/critical → fail после grace-периода 14 дней.
License scan license-checker weekly cron Блок GPL/AGPL в production-зависимостях.

12. План работ

День 1 · ~4ч

Этап 1. Lint, Format, Hooks локально

  • Установить ESLint 9 + typescript-eslint + Prettier 3.
  • Написать root eslint.config.mjs, наследовать в backend и frontend.
  • Настроить Husky + lint-staged + commitlint.
  • Локально прогнать npm run lint и npm run format, починить накопленный шум (обычно ≤ 200 правок).
  • Закоммитить отдельным PR chore(infra): introduce eslint, prettier, husky.
День 2 · ~4ч

Этап 2. CI основной workflow

  • Создать .github/workflows/ci.yml с jobs: lint, typecheck, prisma, backend-tests, frontend-tests (заглушка пока), docker-build, gitleaks.
  • Включить branch protection: required checks + linear history.
  • PR ci(infra): add github actions baseline.
День 3 · ~4ч

Этап 3. Frontend tests + Playwright

  • Установить Vitest + @testing-library/react + jsdom.
  • Написать smoke-тесты на shared/ui (StatusPill, Skeleton, ConfirmDialog, Toast).
  • Установить Playwright, написать E2E «login → dashboard» (используя демо-учётку).
  • Включить frontend-tests job в CI и сделать required.
  • PR test(frontend): vitest + playwright baseline.
День 4 · ~3ч

Этап 4. Renovate, CodeQL, release-please

  • Поставить Renovate-app на репозиторий, добавить renovate.json.
  • Создать codeql.yml, проверить, что Security tab отчитывает.
  • Настроить release-please (включая manifest для workspaces).
  • PR ci(infra): renovate, codeql, release-please.
День 5 · ~3ч

Этап 5. Polishing

  • Добавить бейджи CI в README.
  • Написать backend/prisma/README.md с policy миграций.
  • Создать CONTRIBUTING.md с правилами PR/commit/branch.
  • Описать процесс «локально как в CI»: npm run verify прогоняет всё.
  • PR docs: contributor handbook + ci badges.

13. Acceptance criteria

  • Любой PR в main прогоняет CI; merge заблокирован, если хоть один required-check красный.
  • Локально работают команды: npm run lint, npm run format, npm run format:check, npm run typecheck, npm run verify (агрегатор).
  • Husky блокирует коммит с lint-ошибкой и с невалидным commit-message.
  • Все существующие 7 e2e suites и 3 gateway specs запускаются в CI и зелёные.
  • Vitest + Playwright настроены, минимум 5 unit-тестов на shared/ui и 1 E2E (auth flow).
  • Renovate создает первый еженедельный PR с обновлениями dev-deps; security-alerts активны.
  • CodeQL первый раз запустился, отчёт в Security tab пуст или с known issues.
  • release-please сгенерировал CHANGELOG-PR при первом merge feat-коммита.
  • В README добавлены бейджи: CI status, CodeQL, License, Node version.
  • Pre-commit ловит секреты (smoke-тест: попытка закоммитить JWT — блок).

14.2. Риски

Риск Митigation
Лавина ESLint-ошибок при включении правил → большой «cleanup PR». Включать правила инкрементально: сначала warn, потом error. Использовать --fix и autofix-friendly правила.
CI «висит» из-за медленных e2e на холодном Postgres. Postgres как service-container, --health-*, кеш npm + cache на docker buildx.
Husky тормозит коммит на больших diff. lint-staged по умолчанию только на staged-файлах; pre-push для тяжёлого typecheck.
Renovate шлёт по 30 PR/неделю. Группировка пакетов (nestjs/prisma/frontend core), schedule ночное окно, auto-merge dev-deps.
CodeQL ложноположительные срабатывания. Конфигурировать .github/codeql/codeql-config.yml с suppressions для тестов и моков.
Расхождение Node version между разработчиками. .nvmrc, engines в package.json, Volta-readiness опционально.
Definition of Done CI зелёный на main, branch protection активна, в README — badges, в репо — eslint/prettier/husky/renovate/codeql/release-please. Команда коммитит по Conventional Commits. Frontend имеет минимум 1 e2e и 5 unit тестов. Любой PR проходит через required checks.
