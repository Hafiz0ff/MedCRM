MedCRM · ТЗ №8 · Integrations Hub & Compliance

HL7/FHIR/DICOM, лаборатории, ОМС, фискальная касса, госуслуги

Превратить MedCRM из «острова» в часть медицинской экосистемы. Двусторонний обмен с лабораториями (HL7/LIS), радиологическими станциями (DICOM PACS), кассами (Атол / Эвотор / Soliq), страховыми (ОМС / ДМС), государственными сервисами идентификации (ЕСИА в РФ, eGov в УЗ, единый ID РТ), бухгалтерским ПО (1С / SAP / Bitrix), маркетплейсами (DocDoc, Сберздоровье, BeWell). Реализация через integrations-service с adapter-pattern и outbox/inbox-based reliability.

1. Назначение

MedCRM сейчас умеет только экспортировать данные в FHIR-формате; вход — закрыт. Этого недостаточно, чтобы обслуживать сети клиник, где лаборатория, радиология, страховщики и фискальная касса — отдельные системы. ТЗ №8 описывает построение Integrations Hub — единого сервиса, через который MedCRM двусторонне общается со всеми внешними системами.

Контракт hub:

  • Каждая внешняя система — отдельный adapter с одинаковым lifecycle (configure → connect → sync → reconcile → disconnect).
  • Все исходящие сообщения идут через outbox (ТЗ №4); входящие — через inbox с дедупликацией.
  • Конфигурация per-tenant: можно подключить YouTube-аналитику клиники А и Helix-LIS для клиники Б независимо.
  • Полная observability: каждое сообщение видно в админ-UI с retry, replay, manual override.
Принцип «Один integrations-service, много adapters»

Не плодим N разных сервисов на каждую интеграцию — это переиспользует общий paint reliability/observability/UI.

3. Цели и Out of Scope

Цели

  • Создать integrations-service (ТЗ №6) как единый hub для всех external интеграций.
  • Adapter-pattern: каждый интегратор — реализация общего IntegrationAdapter интерфейса.
  • FHIR R5 read/write (Patient, Practitioner, Encounter, Condition, Observation, MedicationRequest, AllergyIntolerance, ServiceRequest, DiagnosticReport).
  • HL7 v2: ADT^A01/04/08/40, ORM^O01, ORU^R01, MDM^T02.
  • DICOM: Bridge к Orthanc; viewer-link OHIF.
  • LIS-адаптеры: ≥ 3 в РФ (Helix / Invitro / СитиЛаб), ≥ 2 в РТ/УЗ.
  • Фискальная касса: ≥ 2 в РФ (Атол / Эвотор), ≥ 1 в УЗ (Soliq).
  • ОМС: реестр услуг + отчётность.
  • ЕСИА (РФ) для авторизации пациента в portal (ТЗ №5).
  • 1С: справочники + документы (счёт-фактура).
  • Маркетплейс sync (хотя бы DocDoc).
  • Outbox + inbox с retries, DLQ, idempotency.
  • Admin-UI: статус каждой интеграции, манyal replay, history of messages.
  • Per-tenant конфигурация + per-tenant credentials в KMS (ТЗ №3).

Out of Scope

  • Полная сертификация HL7 (это long-running compliance procedure).
  • SNOMED CT subscription (отдельная лицензия).
  • Радиологический workflow (DR/MWL/MPPS) — отдельная задача.
  • Госуслуги «для врача» (ЛНР по СМП) — отдельные ТЗ.
  • EDI с фарм-дистрибьюторами (заказы лекарств) — отдельная программа.
  • Apple HealthKit / Google Fit — фаза 2.

4.3. Adapter interface

export interface IntegrationAdapter<TConfig = unknown> {
  readonly kind: string;
  validateConfig(config: unknown): TConfig;
  testConnection(config: TConfig, secrets: Secrets): Promise<void>;
  sendOutbox(message: OutboxMessage, ctx: AdapterContext): Promise<{ externalId?: string }>;
  receiveInbox(payload: unknown, ctx: AdapterContext): Promise<InboxMessage[]>;
  buildAdminUi(): AdapterAdminUi;        // metadata для UI-конфигуратора
}

export interface AdapterContext {
  tenantId: string;
  providerId: string;
  logger: Logger;
  secrets: Secrets;
  prisma: PrismaClient;
  publishDomainEvent: (subject: string, data: unknown) => Promise<void>;
}

4.4. Конфигурация per tenant

  • Admin-UI шаг 1: выбрать тип adapter → шаг 2: ввести конфиг (поля из buildAdminUi()) → шаг 3: ввести credentials (отправляются в KMS, в БД лежит только ref).
  • testConnection вызывается до сохранения; ошибка показывается в UI.
  • Каждый adapter имеет capabilities (read, write, realtime) — UI это показывает.

5. FHIR R5

5.1. Цели

  • RESTful FHIR R5 server endpoints под /fhir/r5/<Resource>.
  • Auth: SMART-on-FHIR (OAuth2 + scopes) и mTLS для корпоративных partners.
  • Resources phase 1: Patient, Practitioner, Encounter, Condition, Observation, MedicationRequest, AllergyIntolerance, ServiceRequest, DiagnosticReport, Appointment.
  • Read, create, update, search (по common params), $everything для Patient bundle.

5.2. Маппинг

  • Mapper-functions toFhir.<Resource> и fromFhir.<Resource> в integrations-service/fhir/mappers.
  • Используем структуру из ТЗ №7 (структурированные диагнозы/назначения/vitals).
  • Identifier extension: system="https://medcrm.app/tenants/<tenantId>/patient-mr".
  • Locale: text labels мультиязычные через Coding.display.

5.3. CapabilityStatement

5.4. Subscriptions

  • FHIR Subscription R5 (topic-based) для push-уведомлений partner-systems.
  • Webhook (REST hook) и WebSocket каналы.

6. HL7 v2

6.1. Поддерживаемые сообщения

ТипНазначениеПоток
ADT^A01Admit / register patientOutbound: уведомить стационар; Inbound: импорт пациента из больницы
ADT^A04Register outpatientOutbound
ADT^A08Update patient demographicsInbound: получить обновлённые данные
ORM^O01Order messageOutbound: заказ в лабораторию
ORU^R01Result messageInbound: результат анализа
MDM^T02Document messageInbound: импорт outside-документа
SIU^S12Schedule notificationOutbound в стационар; Inbound с записи

6.2. Транспорт

  • MLLP (Minimum Lower Layer Protocol) — стандарт HL7 поверх TCP.
  • HTTP-вариант (для современных LIS).
  • SFTP-batch (для старых систем).
  • Outbox-relay поддерживает все три.

6.3. Парсинг

  • Библиотека hl7-standard или собственный лёгкий парсер для основных типов.
  • Валидация: required segments, encoding chars, message version.
  • ACK^AA / ACK^AE / ACK^AR с message-control-id.

7. DICOM / PACS

  • Используем Orthanc как PACS-сервер (open-source, REST API).
  • integrations-service общается с Orthanc через REST + DICOMweb (QIDO/WADO/STOW).
  • Workflow: радиолог делает снимок → modality (КТ/УЗИ/МРТ) отправляет в Orthanc → callback в MedCRM (через webhook) → создаётся RadiologyReport stub → линк на viewer.
  • Viewer: OHIF Viewer (open-source, embeded iframe или внешний URL) с подписанным токеном.
  • DICOM modality worklist (MWL) — отдельная задача, отметка «есть план».
  • Анонимизация для исследовательских export через Orthanc anonymizer.

8. Лабораторные интеграции

8.1. Стратегия

Лаборатории делятся на два класса:

  • Public (Helix, Invitro, СитиЛаб, Lykon, Kasta) — у них есть API-портал партнёра.
  • Local hospital LIS (1С-Медицина, qMS, MedTrak) — HL7 v2 / REST.

8.2. Adapter для Helix (пример)

  • Конфиг: API-ключ, partner-id, base-url, callback-url.
  • Outbound: создание заказа — POST /orders с структурой {patient, services[], priority, comment}.
  • Inbound: webhook /portal/v1/webhooks/integrations/helix с результатом → парсинг → LabReport+LabResultItem (ТЗ №7).
  • Reconciliation: poll-job каждый час по неполным заказам.

8.3. Generic LIS через HL7 ORU^R01

  • Принимаем по MLLP или HTTPS-endpoint.
  • Парсер ORU → LabReport.
  • Mapping LOINC-кодов (если приходят) → справочник из ТЗ №7.

8.4. UI заказа

  • В ServiceOrder (ТЗ №7) врач выбирает «отправить во внешнюю лабораторию» → диалог выбора провайдера, услуг (mapped к panel), приоритета.
  • После отправки — статус «отправлено в Helix #ORD-123456».

9. Фискальная касса

9.1. Атол (РФ)

  • Веб-сервер Атол Online или локальное API на ККМ.
  • Создание чека после Payment.status=CAPTURED: POST /chek с массивом услуг (наименование, количество, цена, ставка НДС).
  • Получаем fnNumber, fdNumber, fpd, время фискализации.
  • Сохраняем в Receipt + ссылку на ОФД.
  • Отправляем в email/Telegram пациенту.

9.2. Эвотор (РФ)

  • Smart-terminal с CloudCore API.
  • Бизнес-модель: касса в холле клиники работает оффлайн; sync через cloud.

9.3. Soliq (УЗ)

  • Подключение через Soliq REST API.
  • Поддержка фиск. отчётности по требованию ГНК Узбекистана.

9.4. РТ

  • Локальные ККМ; интеграция через CSV-batch report (на старте).
  • API-режим — по мере доступности.

10. ОМС / ДМС

  • ТФОМС (РФ): реестр услуг (R-формат) ежемесячно, реестр счёт-фактур, persona-mapping.
  • ДМС (РФ): SOAP/REST API страховщиков (СОГАЗ, Альфа, Ингосстрах, Ресо).
  • Workflow: пациент с ДМС-полисом → проверка полиса через API страховщика → гарантийное письмо → счёт направляется страховщику.
  • Сохранение страховых полисов: PatientInsurancePolicy с провайдером, номером, сроком.
  • Лимиты по полису, перечень покрываемых услуг — кэш из API.
  • Отчётность XML-форматы по требованиям ТФОМС/страховщика.

11. Госуслуги и ID

11.1. ЕСИА (РФ)

  • OAuth2 / OpenID Connect через ЕСИА.
  • В patient-portal (ТЗ №5) появляется кнопка «Войти через Госуслуги».
  • После авторизации — заполнение профиля из официальных данных (ФИО, дата рождения, СНИЛС).
  • СНИЛС используется как федеральный идентификатор пациента.

11.2. eGov (УЗ)

  • One-ID Узбекистана через OIDC.
  • Маппинг ПИНФЛ.

11.3. РТ единый ID

  • Готовность к интеграции, когда api станет доступным.
  • На старте — телефон + OTP остаётся основным методом auth.

12. Бухгалтерия (1С)

  • 1С Бухгалтерия / Управление торговлей / Медицина — через REST/OData или COM-обмен.
  • Объекты:
    • Справочники: контрагенты (пациенты, страховщики), услуги, врачи (как сотрудники).
    • Документы: счёт-фактура, акт оказанных услуг, поступление денежных средств.
  • Двусторонний sync: новые услуги — в 1С, поступления — обратно в MedCRM (для сверки).
  • Расписание: ежедневная выгрузка реестра услуг в 1С.

13. Маркетплейсы

  • DocDoc / Sberzdorovie / BeWell: партнёрские API.
  • Sync слотов: outbound — наши доступные слоты → marketplace.
  • Sync записей: inbound — пациент записался через marketplace → создаётся Appointment.
  • Sync пациентов: создание пациента (минимальный профиль), линковка по телефону.
  • Sync оплат: marketplace отчитывается о платеже.
  • Webhook-callbacks + reconciliation jobs.

14. Reliability и observability

  • Outbox: все исходящие — через IntegrationOutbox (расширение ТЗ №4).
  • Inbox: все входящие — через IntegrationInbox с дедупликацией по (providerId, externalId).
  • Retry: exponential backoff (1m, 5m, 15m, 1h, 6h, 24h), max 8 попыток, потом DLQ.
  • DLQ-UI: admin может посмотреть, replay, edit payload, mark as ignored.
  • Idempotency: каждое сообщение имеет внешний и внутренний ID.
  • Webhooks security: HMAC-подпись + replay-protection (timestamp window).
  • Rate limits: per-provider quota (например, Helix 100 req/min).
  • Tracing: каждое сообщение имеет traceId, который сквозь outbox → adapter → external system → callback (через correlation header).
  • Alerts: DLQ size > 0, error rate per provider > 5%, last sync > 1 час назад.
  • Admin-UI: list of providers, status, last sync, queue depth, retry, replay, configure.

17. Acceptance criteria

  • integrations-service работает; CRUD providers через admin-UI; testConnection валидирует.
  • Outbox-relay отправляет с exponential backoff; DLQ-UI работает; manual retry/replay доступны.
  • Inbox дедуплицирует по (providerId, externalId).
  • FHIR R5 server: 10 ресурсов, search, $everything, CapabilityStatement, SMART-on-FHIR.
  • HL7 MLLP server принимает ADT/ORM/ORU с ACK; парсер устойчив.
  • Orthanc + OHIF работают; viewer открывается из карточки EMR.
  • ≥ 3 лаб-адаптера (Helix/Invitro/СитиЛаб) или Generic HL7 ORU дают LabReport.
  • ≥ 2 фискальных адаптера (Атол/Эвотор) формируют Receipt с fpd/fn/fd.
  • ЕСИА login для portal реализован.
  • 1С sync: справочники + документы поступления.
  • ≥ 1 marketplace sync расписания работает.
  • Webhooks signed (HMAC) + timestamp window.
  • Tracing сквозной (traceId через outbox → adapter → callback → inbox).
  • Метрики Prometheus per provider + Grafana дашборд.
  • Документация в docs/integrations/*.md для каждого adapter.
