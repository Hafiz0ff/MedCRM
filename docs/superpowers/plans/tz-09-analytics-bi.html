MedCRM · ТЗ №9 · Analytics, BI & Data Warehouse

Аналитический слой: ClickHouse, dbt, дашборды, ML-инсайты

В Prisma уже сидят DwFact*, DoctorKpiMetric, FinancialDailyAggregate, MarketingFunnelMetric. Но никто из них ничего не строит — расчёты разрозненные, дашбордов нет, предиктивных моделей тоже. ТЗ №9 описывает построение полноценного analytics-стека: ClickHouse как DWH с CDC-репликацией из Postgres, dbt-модели для финансов/операций/маркетинга/клиники, дашборды (Metabase / Superset), ML для no-show prediction, executive-дайджесты в Telegram.

1. Назначение

Сегодня в MedCRM есть таблицы для агрегатов, но нет ни систематического ETL, ни дашбордов, ни ML. Владелец клиники не может в одно касание увидеть: какова загрузка кабинетов сегодня? Какой DSO у дебиторки? Сколько потеряно денег на no-show в прошлом месяце? Какие маркетинг-каналы конвертируют? Сколько повторных обращений у каждого врача за квартал?

ТЗ №9 закрывает эту дыру: analytics-service (ТЗ №6) становится владельцем read-models; параллельно строится ClickHouse DWH через CDC-репликацию; на DWH работают dbt-модели и BI-дашборды; ML-сервис делает предсказания (no-show, churn, forecasting).

Принцип «Аналитика — отдельный слой, не нагружающий operational DB»

Никаких сложных аналитических запросов к Postgres-операционке. Всё, что для BI — в ClickHouse, обновляется через CDC + dbt.

3. Цели и Out of Scope

Цели

  • ClickHouse как DWH с CDC-репликацией из Postgres через Debezium / pgoutput.
  • dbt-модели: staging → intermediate → marts (finance, operations, marketing, clinical).
  • Аналитический сервис analytics-service (ТЗ №6) с REST API /api/v1/analytics/*.
  • ≥ 8 встроенных дашбордов в admin-UI MedCRM (через iframe Metabase/Superset + native panels).
  • Self-service BI: Metabase или Superset с tenant-aware row-level security.
  • ML-models v1: no-show prediction, patient churn risk, revenue forecasting.
  • Executive digest: ежедневный/еженедельный отчёт в Telegram/email.
  • Marketing attribution: lead → visit → revenue с UTM-tracking.
  • Anomaly detection на ключевых метриках (резкий рост cancellations и пр.).
  • Tenant isolation: каждый тенант видит только свои данные; cross-tenant аналитика — только для платформы.
  • Историзация: SCD2 для медленно меняющихся справочников (врачи, услуги).

Out of Scope

  • Полная замена operational Postgres ClickHouse-ом.
  • Real-time streaming SQL (Materialize / RisingWave) — фаза 2.
  • Data science notebook environment для пользователей — отдельно.
  • Custom dashboards drag-n-drop для конечного user — используем готовый BI tool.
  • Federated query / data mesh — слишком рано.
  • Apple Health / Google Fit агрегаты — отдельно.

5.1. Технология

Debezium + JDBC sinkПрямо в ClickHouse, без Kafka.Connector ClickHouse менее зрелый.
PeerDB (open source)Лёгкий, лучше всего для Postgres → ClickHouse, кратко-fashioned.Молодой продукт.

Решение PeerDB для прямого Postgres → ClickHouse (логическая репликация на pgoutput, низкий лаг, лёгкая эксплуатация). Опционально Debezium + Kafka, если потребуется fan-out в дополнительные системы.

5.2. Какие таблицы реплицируем

  • Все таблицы из scheduling, emr, finance, communications, inventory, auth.tenants, auth.users, auth.roles, справочники.
  • НЕ реплицируем: encrypted_*-колонки PHI (см. ТЗ №3) — на стороне DWH работаем с хэшами/масками; для full unsafe — отдельный security-controlled load.

5.3. Schema mapping

  • Postgres schemas → ClickHouse databases (scheduling_stg, emr_stg, ...).
  • UUID → UUID в ClickHouse.
  • JSONB → String с tag-engine JSON (или Tuple для известной структуры).

6. ClickHouse DWH

6.1. Развёртывание

  • ClickHouse в docker compose (dev) и K8s (prod, ТЗ №10).
  • Single-node на старте; шардирование при росте.
  • Backup: BACKUP DATABASE в S3/MinIO, retention 30 дней.

6.2. Storage strategy

  • Staging (stg.*): сырые реплицированные таблицы, ReplacingMergeTree по PK.
  • Intermediate (int.*): joined, denormalized, type-corrected. MergeTree partitioned by month.
  • Marts (mart.*): bisness-ready, aggregated. SummingMergeTree / AggregatingMergeTree.

6.3. Tenant isolation

  • Каждая таблица содержит tenant_id.
  • ClickHouse RBAC: per-tenant role, row policy на tenant_id = currentUser-tenant.
  • Для BI-инструмента — тенант пробрасывается через JWT/parameter в подключение.

6.4. Sample table

CREATE TABLE int.fact_appointment (
  appointment_id     UUID,
  tenant_id          UUID,
  patient_id         UUID,
  doctor_id          UUID,
  specialty_code     LowCardinality(String),
  branch_id          UUID,
  start_at           DateTime64(3),
  end_at             DateTime64(3),
  status             LowCardinality(String),
  is_no_show         UInt8,
  is_online_booking  UInt8,
  is_first_visit     UInt8,
  channel            LowCardinality(String),  -- portal | offline | marketplace
  revenue_cents      Int64,
  cancel_reason      LowCardinality(Nullable(String)),
  created_at         DateTime64(3),
  updated_at         DateTime64(3),
  _version           UInt64
)
ENGINE = ReplacingMergeTree(_version)
PARTITION BY toYYYYMM(start_at)
ORDER BY (tenant_id, start_at, appointment_id)
SETTINGS index_granularity = 8192;

7.2. Schedule

  • Incremental-материализация — каждый час по большинству март.
  • Full refresh — раз в неделю на специальных моделях.
  • Trigger через worker (ТЗ №4): analytics.dbt-run job в BullMQ.

7.3. Tests

  • not_null, unique, accepted_values на ключевых атрибутах.
  • relationships (FK-целостность между marts).
  • Custom: «revenue per day > 0 в рабочие дни», «no-show rate ≤ 30%», alarm если нарушение.

10. ML / predictive

10.1. Архитектура

  • Отдельный ml-worker в Python (FastAPI) — модель обучается раз в день, online-inference через REST.
  • Обучающие данные: ClickHouse → pandas/polars dataframe.
  • Модели лёгкие: scikit-learn (LogReg, GradientBoosting), Prophet для forecasting.
  • Tracking: MLflow (single-node) для версионирования моделей.
  • Сервинг: FastAPI /predict с input-валидацией Pydantic.

10.2. Модель v1: No-show prediction

  • Признаки: история no-show пациента, расстояние до клиники (если есть), время до приёма, время суток, день недели, канал записи, цена, специальность, погода (опционально), наличие предоплаты.
  • Целевая: is_no_show в исторических данных.
  • Метрики: ROC AUC ≥ 0.75, precision @ recall=0.5.
  • Применение: для записей в окне 48 часов считаем риск; если > threshold → дополнительный SMS/звонок, предложение предоплаты (ТЗ №5).

10.3. Модель v2: Churn risk

  • Признаки: дни с последнего визита, частота визитов, тип специальностей, средний чек, активность в портале (ТЗ №5).
  • Целевая: пациент «отвалился» (no visit ≥ 365 days).
  • Применение: маркетинг-сегмент «risk of churn» → push-кампания.

10.4. Модель v3: Revenue forecast

  • Prophet на daily revenue с учётом сезонности (неделя, год, праздники).
  • Per-branch и per-specialty форки.
  • Visualization в forecast-дашборде.

10.5. Anomaly detection

  • Простой Z-score + ARIMA-residual.
  • Метрики: no-show rate, cancellation rate, revenue, payment success rate.
  • Alert через ТЗ №4 в Telegram владельцу.

11. Operational realtime

Не всё надо считать через DWH; прямо-сейчас-метрики работают на operational Postgres + Redis:

  • Live-queue (сколько ждёт в холле) — Postgres + Socket.IO.
  • Сегодняшняя выручка — Redis-cached aggregator (worker, ТЗ №4) с обновлением каждые 60s.
  • Текущая загрузка кабинетов — то же.
  • Прочие «исторические» метрики — ClickHouse.

12. Data governance

  • PII / PHI handling: в ClickHouse PHI-колонки реплицируются в encrypted виде (см. ТЗ №3); расшифровка возможна только в специальном secure-режиме для аудиторских отчётов.
  • Аналитические снапшоты: для дашбордов используем только safe-fields (ID, агрегаты, hashed names).
  • Tenant isolation: каждый запрос в BI-tool проходит через connection с per-tenant role + row-policy.
  • Audit: все BI-queries логируются (кто-что-когда), доступ к данным аудитируется (требования ФЗ-152).
  • Retention: cold storage в S3/MinIO для данных старше 7 лет.
  • Data dictionary: dbt docs автогенерируется + публикуется на docs/analytics/.

13. Экспорты и отчёты

  • Executive digest: ежедневный Telegram в 09:00 владельцу: выручка вчера, бронирования сегодня, no-show, alerts.
  • Weekly business review: каждый понедельник по email, PDF с топ-метриками.
  • ТФОМС отчёт — XML батч (ТЗ №8).
  • Бухгалтерия 1С: ежедневный реестр услуг (ТЗ №8).
  • Custom export: пользователь через Metabase может экспортировать CSV/XLSX любого дашборда.
  • FHIR DiagnosticReport bundle: для интеграции с госуслугами (через ТЗ №8).

16. Acceptance criteria

  • ClickHouse в проде; PeerDB CDC лаг ≤ 10s P95.
  • dbt project с ≥ 30 моделями (staging + intermediate + marts по 4 категориям).
  • dbt tests прогоняются в CI; failure блокирует deploy.
  • analytics-service отдаёт KPI per tenant через REST API.
  • ≥ 8 native дашбордов работают в admin-UI MedCRM.
  • Metabase self-service BI доступен из admin-UI; tenant-isolation работает.
  • ML no-show prediction: ROC AUC ≥ 0.75 на test set; интегрирован в booking-flow.
  • Churn model генерирует сегмент пациентов с risk; маркетинг может targeted-кампанию (после ТЗ №4).
  • Revenue forecast на 14 дней с MAPE ≤ 15%.
