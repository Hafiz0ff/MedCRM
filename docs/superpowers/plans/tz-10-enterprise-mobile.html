1. Назначение

После ТЗ №1–9 MedCRM — рабочая, безопасная и осмысленная МИС. Но чтобы продаваться сетям клиник, частным врачам и быть «настоящим SaaS-продуктом», нужно закрыть последний пласт: enterprise-зрелость (K8s, SSO, white-label, биллинг), географическое масштабирование (multi-region РФ/РТ/УЗ/EU с резиденцией данных) и мобильные клиенты для пациентов и врачей.

Это сразу превращает MedCRM из «B2B software для клиники» в «SaaS-платформу для медицинской отрасли»: подписка, marketplace модулей, мобильные приложения, видеоконсультации и подпись документов УКЭП — всё, что корпоративный покупатель ждёт по умолчанию.

Принцип «Production-grade с первого дня, без переписывания»

Helm-чарты и K8s — на базе ТЗ №6 модульного split; mobile — поверх portal/EMR API; биллинг — отдельный модуль; всё опционально, ничто не ломает legacy-инсталляции.

3. Цели и Out of Scope

Цели

  • Helm-чарты для всех сервисов (ТЗ №6); один umbrella chart medcrm.
  • K8s-ready: liveness/readiness probes, graceful shutdown, secrets via ExternalSecrets, HPA.
  • Multi-region routing: edge-домен → ближайший region; sticky session.
  • Data residency: tenant attribute region, выбирается при онбординге; перемещение между регионами — отдельный flow.
  • SSO: SAML2 + OIDC, поддержка Keycloak, Azure AD, Google Workspace, Okta.
  • SCIM 2.0 для user provisioning.
  • White-label: custom domain (через CNAME), TLS managed (Let's Encrypt / Caddy), theme tokens (ТЗ №2) per tenant, logo, email-from.
  • SaaS-биллинг: subscription plans, usage-metered overages, prorating, invoice/receipt automation; провайдеры РФ + Stripe для EU.
  • Marketplace модулей: тенант включает / отключает модули; per-module pricing; preview-trial.
  • Native iOS + Android приложения для пациента (Expo).
  • Native iOS + Android приложения для врача (Expo) с offline-first sync.
  • Telehealth: видеоприём через LiveKit, ICE-config, recording (opt-in), chat + screen-sharing.
  • УКЭП-подпись для legal-документов в РФ.
  • SLA-mode: 99.9% standard, 99.95% enterprise; HA-deployment; PITR backup.
  • DR: full backup + restore drill quarterly; cross-region failover documented.

Out of Scope

  • Полный SOC 2 / HIPAA audit — отдельная программа.
  • Public app store releases с медицинской сертификацией (это другая дисциплина) — выпускаем как «бизнес-tool», не как SaMD.
  • Hardware integration (медицинские приборы напрямую) — отдельный модуль.
  • Wearables (Apple Health/Google Fit) — фаза 2.
  • White-label мобильных приложений (per-brand binary) — отдельная программа.
  • Полная offline-операция backend (only mobile offline) — другая задача.
  • Перевод на multi-tenant БД-кластер с physical isolation — enterprise-by-request only.

4.2. Стандартные элементы каждого chart

  • Deployment с rolling update (maxSurge: 1, maxUnavailable: 0).
  • Service (ClusterIP, headless для StatefulSet).
  • HPA на CPU + custom-metrics (queue depth для workers).
  • PodDisruptionBudget minAvailable: 1.
  • Liveness/Readiness: /health/live, /health/ready endpoints в каждом сервисе.
  • Graceful shutdown через preStop + sigterm handler в Nest.
  • NetworkPolicy: разрешён трафик только между нужными сервисами.
  • SecurityContext: runAsNonRoot, readOnlyRootFilesystem.

4.3. Stateful компоненты

  • Postgres → managed (RDS / Cloud SQL / Yandex managed) предпочтительно; альтернатива — оператор (CloudNative-PG / Zalando).
  • Redis → managed или Redis Operator.
  • MinIO → managed или K8s-operator.
  • ClickHouse → Altinity Operator.
  • NATS → NATS Operator.
  • Orthanc → StatefulSet с PVC.

4.4. Secrets и config

  • External Secrets Operator → AWS Secrets Manager / Vault / Doppler.
  • Никаких plain-text secrets в YAML.
  • Config через ConfigMap + env; feature flags через GitOps (Argo CD).

4.5. CI/CD

  • GitHub Actions (из ТЗ №1) собирает docker images, pushes в registry, обновляет Helm-values via PR.
  • Argo CD синхронизирует кластер с git-repo (GitOps).
  • Progressive delivery: Flagger / Argo Rollouts с canary 10% → 50% → 100% и автоматическим rollback по metrics.

5.2. Routing

  • Global DNS (Cloudflare / NS1) с GeoSteering: по IP → ближайший region.
  • Tenant имеет region: при логине gateway проверяет — если запрос пришёл в неправильный region, делает redirect.
  • Custom-domain белой клиники (см. секцию 7) указывает на правильный edge.

5.3. Data residency

  • Каждый region — независимый stack (Postgres, ClickHouse, MinIO).
  • Tenant данные не пересекают границы региона.
  • Платформа-level данные (subscription, billing) — отдельный «platform-control» регион (например, EU).
  • Tenant migration между регионами — отдельный flow с экспорт-импорт и downtime ≤ 1h.

5.4. Cross-region observability

  • Логи / метрики аггрегируются в single observability stack (Grafana Cloud / Loki / Tempo).
  • На-операционные дашборды — региональные; платформенные — global.

6. SSO / SAML / OIDC

  • Поддержка SAML2 (для корпоративных клиник c Active Directory).
  • Поддержка OIDC (для Okta, Google Workspace, Azure AD).
  • Per-tenant конфигурация: tenant-admin загружает SAML metadata XML / прописывает OIDC issuer + client_id.
  • JIT (just-in-time) provisioning: при первом логине user-создаётся в tenant.
  • SCIM 2.0: provider пушит users в MedCRM (create/update/disable).
  • Role-mapping: SAML/OIDC groups → MedCRM roles.
  • 2FA от SSO-провайдера принимается как валидная (skip MedCRM 2FA).
  • SAML signed assertions + AudienceRestriction + valid-until.

7. White-label и custom-domain

7.1. Domain

  • Tenant настраивает CNAME app.clinic.com → cname.medcrm.app.
  • Платформа авто-получает TLS-сертификат через Caddy / cert-manager.
  • Trafic ingress определяет tenant по Host header.

7.2. Branding

  • Per-tenant theme override через CSS-variables (база — ТЗ №2): brand color, accent, logo URL, fonts.
  • Tenant загружает logo (SVG/PNG), favicon, OG-image.
  • Email-from и template-header брендируются: «Иванова клиника» вместо «MedCRM».
  • SMS-sender ID — per tenant (с провайдером).
  • Patient Portal (ТЗ №5) — белая клиника видит только своё имя.

7.3. Limit

  • White-label — только enterprise-тариф.
  • Mobile apps — white-label binaries — отдельный платный сервис (out-of-scope).

12. Offline-first sync

На мобильном приложении врача: данные доступны без сети, изменения накапливаются и синхронизируются. Паттерн взят из репозитория xiewan1230-maker/SaaS, где этот подход уже отработан.

12.1. Stack

  • Локальный store — SQLite через op-sqlite или expo-sqlite.
  • Replication: GraphQL/REST с since-cursor; на сервере у каждой сущности есть updatedAt и version.
  • Outbox-клиент: изменения хранятся локально с idempotency-key, отправляются при reconnect.
  • Conflict resolution: server-wins по умолчанию; для encounter notes — manual merge prompt.

12.2. Sync engine

// client/sync-engine.ts (Expo)
async function syncCycle(): Promise<void> {
  const lastSync = await store.getLastSyncCursor();
  // 1. Pull
  const incoming = await api.pull({ since: lastSync });
  await store.applyIncoming(incoming);
  // 2. Push outbox
  const outbox = await store.getOutboxBatch(50);
  if (outbox.length) {
    const result = await api.push(outbox);
    await store.markPushed(result.acked);
    if (result.conflicts.length) await ui.showConflicts(result.conflicts);
  }
  await store.setLastSyncCursor(incoming.cursor);
}

12.3. Что хранится offline

  • Расписание текущего дня + следующего дня.
  • Пациенты с записями на эти дни.
  • Anamnesis / последние 5 encounters / актуальные allergies / vitals.
  • Шаблоны протоколов (из ТЗ №7).
  • МНН-поисковик (для назначений).

12.4. Безопасность offline-данных

  • SQLite-database зашифрован через SQLCipher с key, derived from device-keychain (защищён biometric).
  • Авторазлогин при потере device-binding.
  • Кэш PHI чистится при logout / по запросу с сервера.

13. Telehealth (LiveKit)

  • LiveKit Cloud или self-hosted для медиа-трафика.
  • Видео + аудио + screen-sharing + chat.
  • Per-appointment room с access-token (LiveKit JWT с TTL = appointment.duration + 30 min).
  • Server-side rooms открываются при appointment.kind = TELEMEDICINE за 10 мин до старта.
  • Запись (opt-in пациента) → загружается в MinIO с tenant-scoped bucket; ссылка в EMR.
  • Quality-monitor: bitrate, packet loss; auto-reconnect.
  • Native iOS / Android intel: ICE-config с TURN для NAT.
  • Согласие пациента на telemed (legal template) → отдельный consent (ТЗ №3).
  • Регулирование РФ/РТ/УЗ — telemed разрешён в рамках первичной консультации с ограничениями; правовая база per region.

14. E-signature УКЭП (Россия)

  • Базовая подпись encounter и легальных документов — WebAuthn (ТЗ №3) для внутреннего trust.
  • Для юридической значимости в РФ (рецепты, акты, отчёты в ОМС): УКЭП (квалифицированная электронная подпись) через КриптоПро или СигнАль.
  • Архитектура: integrations-service adapter «sign.kriptopro»; адаптер получает payload, отдаёт в УКЭП-сервис врача (через локальный плагин CADESCOM), возвращает PKCS#7 / CAdES-XLong-T.
  • Архивирование подписанных документов в S3 + хранение метаданных подписи.
  • Аналог в УЗ: E-IMZO — отдельный adapter с тем же интерфейсом.
  • Доверенная третья сторона (ДТС) для проверки UEC в РФ — опционально.

15. Production observability и SLA

15.1. Стэк

  • Logs: Loki / OpenObserve / CloudWatch.
  • Metrics: Prometheus + Grafana.
  • Traces: Tempo / Jaeger / SigNoz (ТЗ №6).
  • Synthetic monitoring: Checkly / в-house Playwright periodic prober.
  • Error tracking: Sentry (front + back).
  • On-call: PagerDuty / Opsgenie / Telegram bot для small clinics.

15.2. SLA

TierUptimeResponseResolution
Standard (Solo/Clinic)99.9% (≤ 8.76h downtime / год)P1 — 4 рабочих часаBest effort
Network99.9%P1 — 2 рабочих часаP1 ≤ 24h

15.3. Definition

  • Uptime = success rate of GET /health/live на api-gateway за минуту → calculated monthly.
  • Plans downtime (announced ≥ 7 days) — не входит в SLA.
  • Региональные failures — отдельный region SLA.

16. DR, backup, runbooks

  • Postgres PITR: WAL-G / pgBackRest, retention 30 дней; RTO ≤ 1h, RPO ≤ 5 минут.
  • ClickHouse backup: BACKUP DATABASE → S3 ежедневно; restore drill quarterly.
  • MinIO: cross-region replication для документов и DICOM.
  • NATS: stream retention 30 дней; cross-region mirror.
  • Runbooks: восстановление tenant, rollback deploy, region failover.
  • Chaos engineering: ежеквартально — Chaos Mesh с задачами «kill pod», «network partition».
  • Tenant export: GDPR / 152-ФЗ subject-rights — full export of tenant data → ZIP + FHIR-bundle.

18. Этапы работ

19. Acceptance criteria

  • Все сервисы запускаются в Kubernetes через Helm; Argo CD управляет staging и prod.
  • Multi-region setup: RU и EU регионы работают независимо; tenant.region маршрутизирует трафик.
  • SAML2 и OIDC интеграция работает с Keycloak, Azure AD, Google Workspace (≥ 1 успешная customer-pilot).
  • SCIM 2.0 endpoint принимает user-provisioning.
  • Custom domain с TLS работает; theme per tenant.
  • SaaS-биллинг: 5 планов, trial, usage-metering, prorating, dunning emails.
  • Stripe (EU) и Tinkoff/Сбер (РФ) — заданы как payment providers; subscription lifecycle работает.
  • Marketplace UI позволяет включать/отключать модули с feature-flag effect.
  • Mobile patient app собрана и публикуется в TestFlight + Google Play Internal.
  • Mobile doctor app собрана с offline-first sync (SQLite encrypted).
  • Push-уведомления (APNS + FCM) работают.
  • Telehealth: видеоприём end-to-end на web и mobile; запись (opt-in) сохраняется.
  • УКЭП adapter (КриптоПро) подписывает legal-документы; подпись валидируется.
  • E-IMZO adapter для УЗ — рабочий минимум.
  • SLA-документ опубликован; uptime ≥ 99.9% за 90 дней.
  • Backup + restore drill проведён; runbooks написаны и протестированы.
  • Chaos test (kill pod + network partition) не приводит к data-loss.
  • Документация: developer docs, customer admin docs, runbooks.

Offline-sync conflicts.Server-wins + UI-resolve для важных полей; не позволяем offline-create для финансовых операций (only read).
