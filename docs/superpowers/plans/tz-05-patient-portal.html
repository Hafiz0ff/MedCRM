MedCRM · ТЗ №5 · Patient Portal MVP

Личный кабинет пациента: запись, история, оплата

Превратить MedCRM из «внутренней CRM клиники» в полноценную МИС с лицом к пациенту. Patient Portal MVP даёт пациенту возможность записаться онлайн, увидеть историю визитов и медицинские документы, оплатить услуги и управлять согласиями. Это первая фича, которая превращает MedCRM в продукт «для врача и для пациента».

1. Назначение

Сегодня в MedCRM нет публичного интерфейса для пациента. Пациент может узнать о визите только через звонок регистратора, СМС или Telegram-bot (после ТЗ №4). Это «80% продукт»: операционная часть закрыта, но без портала клиника не получает онлайн-трафик, не снижает нагрузку на регистратуру и не предлагает digital-experience, которого ожидают современные пациенты.

Цель ТЗ №5 — построить публичный модуль patient-portal (новый Next.js-app или отдельный сегмент существующего frontend) с регистрацией пациента, онлайн-записью на приём, личным кабинетом (история, документы, рецепты, оплата). Без портала MedCRM остаётся «внутренней CRM», с порталом — становится МИС.

Принцип «Один пациент — один портал, много клиник; идентичность по телефону»

Пациент регистрируется один раз; через тенант-pin (или ссылку от клиники) он привязывается к нужной клинике; одна запись в портале может видеть данные нескольких клиник по grant.

2. Состояние «как есть»

Слой Сейчас Проблема
Patient identity Модель Patient с tenantId, телефоном, паспортом. Нет «портального аккаунта». Пациент не может «войти». Нет связки между записями пациента в разных клиниках.
Online booking В backend есть booking-flow для регистратора, но без публичного эндпоинта. Чтобы записаться — нужно позвонить.
Frontend Один Next.js-app для admin/registry. Нет публичной зоны без auth, нет SEO-страниц клиники.
Payment gateway Модель PaymentGateway, Payment, но реальной интеграции нет. Нельзя оплатить онлайн.
Documents Есть MedicalRecord, EncounterDocument, Prescription. Пациент не имеет доступа.
Consent Модель PatientLegalDocument + DigitalSignature, но flow в портале нет. Согласие подписывается на бумаге.

3. Цели и Out of Scope

Цели (MVP)

  • Регистрация и авторизация пациента (телефон + OTP).
  • Привязка пациента к клинике (или нескольким) через PIN / QR / deeplink.
  • Онлайн-запись на приём: специальность → врач → слот → подтверждение.
  • История визитов: список приёмов, статус, итог.
  • Документы: получить выписку, рецепт, направление (PDF).
  • Оплата онлайн: pay-as-you-go (счёт после визита) + предоплата при онлайн-записи.
  • Уведомления (Telegram / SMS / push) — pull-режим: пациент видит свои.
  • Согласие на обработку ПДн при первом входе.
  • Профиль: ФИО, контакты, аллергии (read-only), управление согласиями.
  • Кнопка «отменить визит» с правилами клиники.
  • Локализация ru/tj/en/uz.
  • Адаптив, поддержка мобильных браузеров без выделенного app.

Out of Scope

  • Native mobile app (iOS / Android) — отдельное ТЗ.
  • Telemedicine (видеосвязь с врачом) — отдельное ТЗ.
  • Чат с врачом — отдельное ТЗ (после appointments-pipeline).
  • Семейные аккаунты (родитель видит детей) — фаза 2.
  • Биометрия / Face ID — после нативных приложений.
  • Loyalty / promo / referral — фаза 2.
  • Интеграция с госуслугами (ЕМИАС, единый ID РТ) — отдельное ТЗ.
  • Интернет-аптека и доставка лекарств — отдельный продукт.

4. Персоны и сценарии

4.1. Персона

«Карина, 32 года, мама двоих детей. Регулярно посещает терапевта, педиатра, гинеколога. Часто записывается в 3 разных клиники. Хочет: всё в одном месте, без звонков, мобильный браузер.»

4.2. Ключевые сценарии

  1. S1 — Первая запись через QR клиники. Сканирует QR в холле → вводит телефон → OTP → подтверждает согласие → видит специальности клиники → выбирает «терапевт» → выбирает доктора → выбирает слот → оплачивает онлайн или «оплатить в клинике» → получает Telegram-confirmation.
  2. S2 — Повторная запись из портала. Вход → «Записаться» → клиника (выбор из привязанных) → доктор → слот → оплата → готово.
  3. S3 — Просмотр истории. Вход → «Мои визиты» → конкретный визит → выписка PDF → скачать.
  4. S4 — Получение рецепта. Уведомление в Telegram «новый рецепт» → клик → авторизация → видит рецепт → скачивает PDF → показывает в аптеке (QR).
  5. S5 — Оплата задолженности. На главной портала видит «у вас счёт на 1 200 ₽» → клик → оплачивает → видит чек.
  6. S6 — Отмена визита. «Мои визиты» → активный визит → «Отменить» → выбор причины → confirm → освобождается слот.
  7. S7 — Перенос визита. «Мои визиты» → активный → «Перенести» → выбор нового слота того же врача → confirm.
  8. S8 — Multi-clinic. Tenant A пациент уже зарегистрирован; tenant B приглашает по deeplink → confirm → один аккаунт видит и A, и B.

5. Архитектура портала

5.1. Опции деплоя

Вариант Описание Pros Cons
A · Один Next.js-app с двумя зонами frontend/app/(portal) и frontend/app/(protected) Один deploy, общие токены, общая дизайн-система. Бандл больше, риск утечки кода между зонами.
B · Отдельный Next.js-app portal/ Новый workspace в монорепо Чёткая граница, отдельный DNS / SEO, меньший бандл портала. Дополнительный deploy, нужна синхронизация дизайн-токенов.
C · SSR в gateway API-only backend, портал на статике с client-side fetch — SEO хуже, для SaaS — нерекомендуется.
Решение Вариант B — отдельный portal/ workspace в монорепо.

Преимущества разделения: чистый бандл (нет admin-кода), отдельный domain (portal.<clinic>.medcrm.app или portal.medcrm.app/t/<tenantCode>), отдельные security headers (более строгие CSP), отдельный CDN и кэширование SEO-страниц «врачи / специальности / отзывы». Token и UI-kit из ТЗ №2 переиспользуются.

5.2. Domain-стратегия

  • portal.medcrm.app/t/<tenantCode> — fallback вариант для любой клиники.
  • <tenantCode>.medcrm.app/portal — стандарт для подписки.
  • Кастомный домен (app.<clinic>.ru) — для enterprise тарифа.

5.3. Backend

Эндпоинты портала — отдельный модуль patient-portal в auth-service (или gateway-уровень):

  • Под отдельным префиксом /portal/v1/*.
  • Auth — отдельный flow (OTP по телефону), отдельные роли (PATIENT).
  • RBAC: пациент может читать только свои данные (по patientId в JWT) + RLS политики (см. ТЗ №3).

6. Auth пациента

6.1. Flow регистрации

Step 1

Phone input

Пациент вводит телефон в формате +992XXXXXXXXX.

Step 2

OTP via SMS / Telegram

Worker отправляет 6-значный код (см. ТЗ №4). Срок 5 минут, max 3 попытки.

Step 3

Verify OTP

Backend проверяет код, выпускает session token (cookie SameSite=Strict).

Step 4

Profile setup

Если новый пациент: ФИО, дата рождения, пол. Иначе skip.

Step 5

Consent

Согласие на обработку ПДн, подписывается digital-signature.

6.2. Привязка к клинике

  1. Через deeplink клиники: /portal/connect?tenant=demo-clinic&token=<hmac> — токен подписан tenant-secret, валидируется backend.
  2. Через QR-код в холле клиники (тот же deeplink, но в виде QR).
  3. Через PIN от регистратора: «введите PIN, который вам сообщили в клинике» (5-значный, TTL 10 мин).
  4. Через приглашение в SMS / Telegram после первого визита.

После привязки запись в PatientPortalGrant:

model PatientPortalGrant {
  id            String   @id @default(uuid()) @db.Uuid
  accountId     String   @map("account_id") @db.Uuid
  tenantId      String   @map("tenant_id") @db.Uuid
  patientId     String   @map("patient_id") @db.Uuid
  state         String   @default("ACTIVE")   // ACTIVE | REVOKED
  grantedVia    String?  @map("granted_via")  // DEEPLINK | QR | PIN | INVITE
  grantedAt     DateTime @default(now()) @map("granted_at") @db.Timestamptz(6)
  revokedAt     DateTime? @map("revoked_at") @db.Timestamptz(6)
}

6.3. Сессия пациента

  • Access token JWT 1h, refresh 60 дней (запоминаем браузер).
  • Cookie HttpOnly + Secure + SameSite=Strict.
  • Devices: модель PatientPortalSession хранит fingerprint, IP, last-active.
  • «Выйти на всех устройствах» — кнопка в профиле.

6.4. Опциональная 2FA

  • TOTP-приложение (после ТЗ №3).
  • Включается пациентом сам в Settings → Безопасность.
  • WebAuthn — фаза 2.

6.5. Восстановление доступа

  • Telephone changed → восстанавливается через клинику (проверка паспорта в офлайне) + одобрение clinic-admin.
  • Lost OTP device → recovery codes (если включена 2FA).

7. Online booking

7.1. Funnel

Step 1

Specialty

Список специальностей клиники с иконками и описанием.

Step 2

Doctor

Список врачей по specialty: фото, опыт, рейтинг (опционально), ближайший слот.

Step 3

Slot

Календарь × 14 дней, доступные слоты; учёт ResourceCalendar.

Step 4

Service & Pre-pay

Выбор услуги (опционально), сумма, опция «предоплата» или «оплатить в клинике».

Step 5

Confirm

Создание Appointment в статусе SCHEDULED, Telegram/SMS-уведомление.

7.2. Учёт слотов

  • Слоты считаются на лету backend-ом: BookingSlotsService.getAvailable(doctorId, dateRange).
  • Учитываются: ResourceCalendar доктора, AppointmentReservation (временный hold), уже забронированные Appointment.
  • Hold-механика: при выборе слота создаётся AppointmentReservation на 5 минут — другие пациенты не могут выбрать тот же слот.
  • При confirm — reservation превращается в Appointment; при таймауте — освобождается.

7.3. Бизнес-правила

Правило Конфигурация tenant
Окно онлайн-записи от now + lead_minutes до now + horizon_days; default 60 минут / 30 дней
Cancel deadline не позже start_at − cancel_grace_hours; default 4 часа
Re-schedule deadline не позже start_at − reschedule_grace_hours; default 4 часа
Pre-pay может быть «опционально», «обязательно», «зависит от специальности»
No-show penalty удержание предоплаты согласно публичной оферте клиники
Max active bookings default 3 одновременно (anti-spam)

7.4. SEO-страницы клиники

  • Public-страница клиники: услуги, врачи, прайс — без auth.
  • Public-страница врача: специализация, опыт, ближайшие слоты, кнопка «записаться».
  • SSR + sitemap.xml + JSON-LD структурированные данные (Schema.org Physician + MedicalClinic).

8.2. Документы пациента

  • Выписка из визита — генерируется из Encounter + EncounterDiagnosis + EncounterDocument в worker (PDF queue).
  • Рецепт — Prescription: PDF + QR-код с подписанной payload (для проверки в аптеке).
  • Направление на анализы — на основе ServiceOrder.
  • Чек — Receipt после оплаты.
  • Доступ — только своему пациенту; signed-URL TTL 15 минут.
  • Логирование доступа к каждому документу (audit.action: 'document_viewed') — см. ТЗ №3.

8.3. Profile

  • ФИО, дата рождения — read-only после verify; изменение через клинику.
  • Телефон — обновляется с reverify через OTP.
  • Адрес — пациент редактирует свободно.
  • Аллергии, хронические — read-only (заполняет врач).
  • Управление согласиями (см. секцию 10).
  • Управление подписками на уведомления.

Provider-абстракция аналогична channel-providers (см. ТЗ №4):

export interface PaymentProvider {
  readonly code: string;
  createPayment(input: { amount: Money; description: string; returnUrl: string; meta: Record<string, unknown> }): Promise<{ paymentId: string; payUrl: string }>;
  capture(paymentId: string): Promise<{ status: 'CAPTURED' | 'CANCELLED'; receipt?: ReceiptData }>;
  refund(paymentId: string, amount?: Money): Promise<{ status: 'REFUNDED' | 'PARTIAL_REFUNDED' }>;
  handleWebhook(payload: unknown, signature: string): Promise<PaymentWebhookEvent>;
}

9.2. Pre-pay flow

  1. Пациент выбирает слот, услугу → backend создаёт Invoice в статусе PENDING + Payment с provider.
  2. Provider возвращает payUrl → пациент идёт на checkout (hosted page).
  3. После успеха provider шлёт webhook → backend помечает Payment.status = CAPTURED, Invoice.status = PAID, Appointment.confirmedAt.
  4. При timeout (15 минут) — Appointment отменяется автоматически, slot освобождается.

9.3. Post-visit pay

  1. Врач завершает приём; счёт формируется в кассе клиники.
  2. Counter / Telegram bot уведомляет пациента: «оплатите <сумма> — ссылка».
  3. Пациент в портале видит «к оплате 1 200 ₽» — клик → провайдер → success.
  4. Backend генерирует фискальный чек (если есть интеграция с фискальной кассой) + PDF чек.

9.4. Refund

  • Cancel в окне разрешения → автоматический refund (полный).
  • Cancel после deadline → клиника решает manually (admin endpoint).
  • Refund фиксируется через PaymentAllocation; audit-log.

9.5. PCI compliance

  • Никогда не храним PAN. Используем hosted checkout / tokenization.
  • В коде только paymentId, last4, brand.
  • Webhook-секрет хранится в KMS (см. ТЗ №3).
  • Signed URLs для receipt PDF; TTL 15 минут.

10. Согласия и приватность

  • При первом входе — обязательное согласие на обработку ПДн (текст из LegalDocumentTemplate).
  • Подпись — DigitalSignature с timestamp, hash; запись в PatientLegalDocument.
  • Управление в Settings: список действующих согласий, дата подписания, кнопка «отозвать».
  • Отозванное согласие закрывает обработку, но клиника обязана сохранить медкарту 7 лет (см. ТЗ №3).
  • Cookie banner на public-страницах (essential + analytics; пациент управляет).
  • Privacy Policy и ToS — обязательные linked pages.
  • Right to access / portability — кнопка «Скачать все мои данные» → backend генерирует ZIP (FHIR Bundle + PDF + CSV).
  • Right to erasure — кнопка «Удалить мой портал-аккаунт»: revoke grants, anonymize portal account; медкарта остаётся за клиникой.

11. Уведомления пациента

  • Reminders по визиту (см. ТЗ №4) идут в Telegram / SMS — портал показывает их в feed.
  • Web Push (Service Worker): новые рецепты, изменения статуса визита.
  • In-app feed «Уведомления» в портале.
  • Подписка на каналы: пациент может отключить marketing, оставить только transactional.
  • Telegram-bot линкуется к porталу через deeplink (одинаковый telegram chatId у пациент-портала и у клинического бота).

12. Изменения в Prisma

// 1. Portal account: один на пациента, независимо от клиники
model PatientPortalAccount {
  id              String   @id @default(uuid()) @db.Uuid
  phoneE164       String   @unique @map("phone_e164") @db.VarChar(20)
  phoneBi         String   @map("phone_bi") @db.VarChar(64)
  emailE164       String?  @unique @map("email_e164") @db.VarChar(255)
  firstName       String?  @map("first_name") @db.VarChar(120)
  lastName        String?  @map("last_name") @db.VarChar(120)
  birthDate       DateTime? @map("birth_date") @db.Date
  gender          String?  @db.VarChar(16)
  twofaEnabled    Boolean  @default(false) @map("twofa_enabled")
  twofaSecretHash String?  @map("twofa_secret_hash")
  createdAt       DateTime @default(now()) @map("created_at") @db.Timestamptz(6)
  updatedAt       DateTime @updatedAt @map("updated_at") @db.Timestamptz(6)

  sessions        PatientPortalSession[]
  grants          PatientPortalGrant[]

  @@map("patient_portal_accounts")
}

// 2. Связь портал-аккаунта с пациентом конкретной клиники
model PatientPortalGrant {
  id            String   @id @default(uuid()) @db.Uuid
  accountId     String   @map("account_id") @db.Uuid
  tenantId      String   @map("tenant_id") @db.Uuid
  patientId     String   @map("patient_id") @db.Uuid
  state         String   @default("ACTIVE") @db.VarChar(20)
  grantedVia    String?  @map("granted_via") @db.VarChar(40)
  grantedAt     DateTime @default(now()) @map("granted_at") @db.Timestamptz(6)
  revokedAt     DateTime? @map("revoked_at") @db.Timestamptz(6)

  account       PatientPortalAccount @relation(fields: [accountId], references: [id], onDelete: Cascade)
  tenant        Tenant   @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  patient       Patient  @relation(fields: [patientId], references: [id], onDelete: Cascade)

  @@unique([accountId, tenantId, patientId])
  @@index([tenantId, patientId])
  @@map("patient_portal_grants")
}

16. Acceptance criteria

  • Пациент регистрируется по телефону + OTP; OTP реально доходит через SMS или Telegram.
  • Пациент привязывается к клинике через deeplink, QR или PIN.
  • Один портал-аккаунт может видеть данные нескольких клиник.
  • Booking funnel рабочий: пациент записывается online, slot блокируется hold-механикой.
  • Пациент может отменить и перенести визит в окне правил клиники.
  • Все «активные» и «прошедшие» визиты видны в истории.
  • Выписка, рецепт, направление — PDF скачиваются по signed URL.
  • QR-код на рецепте валиден и проверяем сторонним сервисом аптеки.
  • Pre-pay flow работает через минимум один реальный провайдер (YooKassa или Alif).
  • Refund при cancel-в-окне выполняется автоматически.
  • Webhook платёжки подписан и валидируется.
  • Согласие на ПДн подписывается и хранится; пациент может отозвать.
  • Export данных (ZIP) и delete portal account работают end-to-end.
  • Web Push доставляет нотификацию о новом рецепте.
  • Локализация ru/tj/en/uz; контент UI без хардкода.
  • Lighthouse: LCP < 2.5s, CLS < 0.1, TBT < 200ms на главной портала и booking-флоу.
  • Audit-log записывает каждый просмотр документа и каждое изменение профиля.
  • Pilot-запуск на demo-клинике завершён, фидбэк собран.

17. Метрики и риски

17.1. Бизнес-метрики

Online-bookings≥ 30%% от total через 3 мес
No-show rate−25%vs offline-only
Pre-pay rate≥ 40%от online-bookings
Activation≥ 60%завершают connect → booking
Phone-load−40%звонков на регистратуру
Re-use≥ 50%пациент делает 2-ю запись через портал
NPS≥ 45после первого пилота
Auth success≥ 95%OTP получен и введён

17.2. Технические метрики

LCP portal≤ 2.5 sP75 4G
Booking ttfb≤ 300 msAPI P95
Payment success≥ 97%от инициированных
Webhook lag≤ 5 sот capture до Invoice.PAID

17.3. Риски

Риск Митигация
Race на одном слоте (два пациента жмут «подтвердить»). DB-уровень: UNIQUE на (doctorId, startAt) с EXCLUDE GIST; reservation hold; SELECT FOR UPDATE в booking-tx.
Платёжка отвечает «captured», но webhook не дошёл. Reconciliation worker: каждые 15 минут поллит status платежа по paymentId, идемпотентно обновляет.
Refund при отмене — провайдер вернул ошибку. Retry + DLQ + админ-uIalert; ручной refund доступен из админ-uI.
SPAM-bookings: один пациент создаёт 100 reservation. Rate-limit per accountId; max_active_bookings default 3; phone-otp требуется для каждой регистрации.
Утечка PHI через portal (например, передал ссылку коллеге). Signed URL TTL 15 мин; audit access; web push только на subscribed device; не отправляем PHI в email/SMS.
Утерян телефон → потерян доступ. Recovery через клинику (admin-flow) с проверкой паспорта; backup-codes (если 2FA включена).
Tenant хочет белый-label портал на своём домене. Поддержка custom-domain в архитектуре с начала; TLS managed via Caddy/Let's Encrypt; tenant-specific theming через токены.
Регуляторное: РФ требует ID госуслуг для медзаписи. Сейчас — phone-OTP достаточно для частной клиники; ЕМИАС/ЕСИА — отдельное ТЗ.
iOS Safari Web Push поддержка ограничена. Fallback на Telegram bot; web push — bonus, не критичен.
Локализация рецепта на узбекском. Шаблоны PDF — TT&T-уровень; перевод проверяется клиникой.
Definition of Done Patient Portal MVP в проде: OTP login работает, booking + cancel + reschedule, история + документы, оплата online через YooKassa и Alif, push-уведомления, согласия и subject rights, локализация ru/tj/en/uz, audit на каждое действие. Пилот в demo-клинике показал ≥ 30% online-bookings и снижение no-show ≥ 20%.
