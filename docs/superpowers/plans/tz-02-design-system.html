1. Назначение

README MedCRM обещает «Next.js 16 · React 19 · TailwindCSS 4 · shadcn/ui · TanStack Query · Socket.IO Client · lucide-react», а архитектурные ТЗ дополняют этот стек Zustand, React Hook Form + Zod, next-intl, next-pwa, ECharts. По факту в frontend/package.json установлены только React, Next, TanStack Query, lucide-react, react-hook-form, socket.io-client, zod, zustand. Tailwind, shadcn/ui, ECharts, next-intl, next-pwa — отсутствуют.

Цель ТЗ №2 — закрыть это расхождение и построить дизайн-систему уровня «продукт, а не прототип»: единые design tokens, библиотека компонентов через shadcn/ui поверх Radix, документация в Storybook, тёмная тема, density-режимы для регистратуры и EMR, i18n RU/TJ/EN/UZ, графики и accessibility.

Принцип «Дизайн-система — это контракт между дизайнером и инженером, формализованный в коде»

Tokens в одном источнике (CSS variables + Tailwind theme), компоненты в Storybook, тесты на каждом компоненте, переиспользование вместо дублирования.

3. Цели и Out of Scope

Цели

  • Установить Tailwind 4 (CSS-first config) и shadcn/ui поверх Radix.
  • Зафиксировать дизайн-токены: цвет, типографика, spacing, radii, shadow, motion.
  • Light + dark + high-contrast темы.
  • Density: compact (live-queue/реестры) и comfortable (EMR/формы).
  • Storybook 8 c документацией всех компонентов и паттернов.
  • i18n (next-intl) с RU как default + TJ/EN/UZ.
  • Charts через Recharts (рекомендация) или ECharts (опция).
  • Iconography: lucide-react (актуальная версия).
  • Forms: единый <Form>+<FormField> поверх react-hook-form + Zod.
  • Accessibility WCAG 2.2 AA на критичных экранах (login, registry, EMR, кассa).
  • Visual regression via Storybook Test Runner или Chromatic.

Out of Scope

  • Полная миграция всех экранов в один спринт — выполняется поэтапно.
  • Native mobile (Expo) — отдельное долгосрочное ТЗ.
  • Подмена существующих CSS-переменных без шага совместимости — выполняем bridge.
  • Marketing-сайт (отдельный домен) — отдельная задача.
  • Email-template-системы (Maizzle/Inky) — отдельная задача.
  • Полный rebrand: логотип/айдентика — отдельная задача дизайна.

4. Архитектурное решение

Tailwind 4

CSS-first config через @theme. Tokens в app/globals.css остаются как CSS-переменные, плюс Tailwind подхватывает их в @theme inline.

shadcn/ui

Компоненты подключаются как owned-code через npx shadcn add. Никакого external runtime; всё в shared/ui/.

Radix Primitives

Headless-фундамент под shadcn: Dialog, Popover, Tooltip, Combobox, Select. Гарантирует a11y из коробки.

Почему не «оставить vanilla CSS» Текущий стиль работает на 6 компонентах, но не масштабируется на 60–80 экранов клиники.

Tailwind 4 + shadcn/ui — индустриальный стандарт 2025–2026, который ускоряет разработку в 3–4 раза за счёт готовых паттернов, утилитарного автокомплита и устойчивого UI-стиля. Подход «owned code» (shadcn копирует код в репо) уменьшает риск vendor-lock.

7. Storybook

Storybook 8 — основная документация UI-kit:

  • Структура stories: shared/ui/<Component>/<Component>.stories.tsx.
  • Globals: theme (light/dark), density (compact/comfortable), locale (ru/tj/en/uz) — через Storybook toolbar.
  • Plugins: @storybook/addon-a11y, @storybook/addon-themes, @storybook/addon-interactions, @storybook/test-runner.
  • Visual regression: Chromatic (опционально) или Playwright-based snapshot tests.
  • Документация написана сразу в stories через autodocs.
  • Deploy: Vercel preview на каждый PR (storybook-static/).

8. Density-режимы и темы

8.1. Density

Два режима, выбираемых через data-density на <html>:

Режим Применение Row height Field height Spacing
compact Live-queue, реестры, расписание, склад 36px 32px ×0.875
comfortable EMR, формы создания, пациент-карта, settings 48px 40px ×1.0

Переключение хранится в localStorage + tenant preference (через Settings API).

8.2. Темы

Тема Когда
light (default) Дневная смена
dark Ночная смена, дежурные врачи, EMR при работе с лучевой
high-contrast WCAG AAA, для слабовидящих пользователей

Переключение: Settings → Внешний вид; respect prefers-color-scheme по умолчанию.

9. i18n / Localization

Подключаем next-intl:

  • Структура: frontend/i18n/{ru,tj,en,uz}.json с namespaces по модулям.
  • Provider в app/[locale]/layout.tsx; default ru.
  • Все строки UI вытаскиваются в JSON; запрет на захардкоженный текст в JSX.
  • Серверные хелперы для письма/SMS — отдельный namespace backend/i18n/ (выходит за рамки этого ТЗ, но важна согласованность ключей).
  • Числа/даты/валюты — через Intl.NumberFormat и Intl.DateTimeFormat; ICU-format в JSON.
  • RTL не требуется (нет арабского), но кодовая база не должна предполагать LTR явно.
  • Локали: ru-RU (default), tg-TJ (Тоҷикӣ), en-US, uz-UZ.

11. Accessibility

  • Target: WCAG 2.2 Level AA на критичных потоках (login, registry, EMR, касса).
  • Inline axe checks в Storybook (@storybook/addon-a11y).
  • Axe-core в Playwright e2e (@axe-core/playwright).
  • Keyboard navigation: Tab/Shift+Tab, Esc закрывает диалоги, Enter активирует primary action.
  • Skip links на каждом layout-shell.
  • Visible focus ring (2px outline brand-600).
  • Контрастность: ≥4.5:1 для текста, ≥3:1 для UI-границ.
  • Screen reader labels на iconic buttons (aria-label).
  • Form errors: aria-describedby + visible message under input.
  • Locale-aware date/number/currency.

14. Migration plan

Подмена визуального стиля выполняется без «бигбэнга»:

  1. Phase 0 — Bridge: существующие CSS-переменные продолжают работать. Tailwind 4 подключается параллельно, @theme inline просто читает текущие --brand, --surface и т.д.
  2. Phase 1 — UI-kit: заменяем компоненты в shared/ui/ на shadcn. Старые остаются как fallback до миграции страниц.
  3. Phase 2 — Page migration: по одному экрану переводим на новые компоненты. Очерёдность: login → dashboard → live-queue → patients → scheduling → EMR → finance.
  4. Phase 3 — Cleanup: удаляем legacy CSS, переключаемся с глобальных классов на utility-классы Tailwind.
  5. Phase 4 — i18n: запускаем next-intl, переводим ru-only код через codemod.
  6. Phase 5 — Dark/Density: включаем темы и density-режимы; smoke-проверка по экранам.
RM499 совместимость В репо уже есть rm499 редизайн.

Перевод компонентов на shadcn должен сохранить визуальный язык rm499: компактные таблицы, белая боковая панель, бирюзовый акцент, dots-статусы. Дизайн-токены пишутся «под rm499».
