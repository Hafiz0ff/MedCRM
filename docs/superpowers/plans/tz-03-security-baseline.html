1. Назначение

MedCRM работает с защищённой медицинской информацией (ПДн пациента, диагноз, рецепт, финансы). Базовая архитектура поддерживает tenantId на каждой таблице и RBAC на уровне приложения, но «hard boundaries» в БД нет — забытый where tenantId в одном запросе означает утечку между клиниками.

Кроме того, в Prisma уже есть модель User2faSettings (с полями isEnabled, secretHash, backupCodes, preferredMethod), но flow в коде backend/apps/auth-service/src/auth/ не реализован. То же по audit-логу: модель AuditLog существует, hash-chain отсутствует.

Цель ТЗ №3 — закрыть фундаментальные дыры безопасности и комплаенса, без которых нельзя продавать MedCRM медицинской организации.

Принцип «Defense in depth»

Tenant isolation должна работать на двух уровнях минимум: в коде (Prisma middleware) и в БД (RLS). 2FA — обязательна для админов. PHI шифруется на колонке. Audit-лог нельзя подделать.

2. Состояние «как есть»

Область Сейчас Дыра
Tenant isolation В каждой Prisma-запрос вручную добавляется tenantId; есть core/tenancy. Забытый фильтр — утечка. Нет защиты на уровне БД.
2FA Модель User2faSettings существует, но эндпоинтов POST /auth/2fa/setup, POST /auth/2fa/verify нет. Админ заходит по логин+паролю.
Sessions JWT access 15m + refresh 30d (см. .env.example), модель UserSession. Нет revoke по device, нет «sign out everywhere», нет risk-based-policy.
Audit Модель AuditLog с oldValues/newValues/requestId; индекс по (tenantId, action, createdAt). Нет hash-chain, нет «append-only» защиты, любой админ БД может изменить лог.
PHI шифрование Поля пациента (имя, телефон, диагноз) хранятся как plaintext. Любой SELECT * из БД даёт сразу PHI.
Секреты JWT_ACCESS_SECRET, JWT_REFRESH_SECRET в .env; change_me_* в .env.example. Нет ротации, нет KMS, риск утечки через бэкап репо.
Rate limiting Gateway имеет rate-limit.middleware с in-memory store. В кластере (≥2 поды gateway) лимиты неконсистентны. Нет защиты на auth-эндпоинтах в auth-service.
Headers Helmet установлен. Нет CSP/Permissions-Policy/HSTS preload-конфигурации.
SAST/DAST Нет. Покрывается ТЗ №1 (CodeQL) и здесь — OWASP ZAP baseline.
Compliance docs Нет. Нет DPA, нет описания DPIA, нет процедуры response на subject request.

3. Threat model (упрощённая STRIDE)

HighT1 · Cross-tenant data leak

Забытый where tenantId или баг в join — клиника A видит данные клиники B.

HighT2 · Account takeover админа

Пароль слит через phishing → доступ ко всей клинике.

HighT3 · PHI exfiltration через БД

Доступ к Postgres (read-only replica, бэкап) даёт plaintext PHI.

HighT4 · Audit log tampering

Админ БД редактирует audit_logs чтобы скрыть удаление пациента.

MedT5 · Session hijacking

XSS / устарелый JWT, нет device fingerprint.

MedT6 · Brute force на /auth/login

In-memory rate limit не работает на кластере.

MedT7 · Privilege escalation

Регистратор отправляет role: admin в payload — gateway не валидирует RBAC.

MedT8 · Insecure file upload

MinIO bucket для медкарт без anti-virus и без подписанных URL.

LowT9 · Outdated dependency

Покрыто Renovate (ТЗ №1).

4. Цели и Out of Scope

Цели

  • Включить PostgreSQL Row-Level Security на всех таблицах с tenant_id.
  • Сделать 2FA TOTP обязательной для админов; backup-codes; опционально WebAuthn для enterprise.
  • Сделать sessions revoke-aware: per-device, «sign out everywhere», risk-based MFA challenge.
  • Превратить audit_logs в append-only с hash-chain.
  • Column-level шифрование PHI для критичных полей (имя, телефон, паспорт, диагноз).
  • Перенос секретов из .env в Doppler / Vault / SOPS.
  • Унификация rate-limit на Redis (gateway + auth-service).
  • Усилить заголовки безопасности (CSP, HSTS, Permissions-Policy).
  • OWASP ZAP baseline в CI как health-check.
  • Подготовка комплаенс-документации (DPA, DPIA, subject request).

Out of Scope

  • SSO (SAML/OIDC) для enterprise — отдельное мини-ТЗ.
  • Полная сертификация HIPAA/ISO27001 — это юридический процесс.
  • Bug bounty программа — после публичного релиза.
  • FIPS 140-2 cryptography hardware — для нишевых клиентов.
  • Pentest вручную — после реализации baseline.

5. PostgreSQL Row-Level Security

5.1. Архитектура

Каждая таблица с колонкой tenant_id получает RLS-политику, проверяющую соответствие сессионной переменной app.current_tenant_id. Соединение с БД через Prisma выполняет SET app.current_tenant_id = '<uuid>' в начале транзакции.

5.2. Миграция (пример для двух таблиц)

-- 20260601100000_enable_rls.sql

-- 1) Helper functions
CREATE OR REPLACE FUNCTION current_tenant_id() RETURNS uuid AS $$
  SELECT NULLIF(current_setting('app.current_tenant_id', true), '')::uuid;
$$ LANGUAGE sql STABLE;

CREATE OR REPLACE FUNCTION current_user_id() RETURNS uuid AS $$
  SELECT NULLIF(current_setting('app.current_user_id', true), '')::uuid;
$$ LANGUAGE sql STABLE;

-- 2) Application roles
CREATE ROLE medcrm_app NOINHERIT LOGIN PASSWORD 'configured-out-of-band';
CREATE ROLE medcrm_migrator NOINHERIT LOGIN PASSWORD 'configured-out-of-band';

-- 3) Enable RLS on tenant-scoped tables
ALTER TABLE patients      ENABLE ROW LEVEL SECURITY;
ALTER TABLE patients      FORCE  ROW LEVEL SECURITY;

CREATE POLICY patients_tenant_isolation
  ON patients
  USING (tenant_id = current_tenant_id())
  WITH CHECK (tenant_id = current_tenant_id());

ALTER TABLE appointments  ENABLE ROW LEVEL SECURITY;
ALTER TABLE appointments  FORCE  ROW LEVEL SECURITY;

CREATE POLICY appointments_tenant_isolation
  ON appointments
  USING (tenant_id = current_tenant_id())
  WITH CHECK (tenant_id = current_tenant_id());

-- ...repeat for all tables with tenant_id
-- Generated by a script that introspects information_schema

5.3. Prisma middleware устанавливает session vars

// backend/core/tenancy/tenant-prisma.middleware.ts
prisma.$use(async (params, next) => {
  const ctx = TenantContext.current();   // AsyncLocalStorage
  if (!ctx) throw new Error('Tenant context missing');

  // Wrap into a transaction with session vars
  return prisma.$transaction(async (tx) => {
    await tx.$executeRawUnsafe(
      `SELECT set_config('app.current_tenant_id', $1, true)`,
      ctx.tenantId,
    );
    if (ctx.userId) {
      await tx.$executeRawUnsafe(
        `SELECT set_config('app.current_user_id', $1, true)`,
        ctx.userId,
      );
    }
    return next({ ...params, runInTransaction: true });
  });
});

5.4. Особые случаи

  • Платформенные таблицы (tenants, permissions, integration_providers) — RLS-политика «по принципу whitelist роли». Только medcrm_platform_admin может читать.
  • Cross-tenant операции (например, billing-агрегация по всем клиникам): отдельный сервис с ролью medcrm_platform_admin; RLS обходить через SECURITY DEFINER функции.
  • Миграции: запускаются под ролью medcrm_migrator (BYPASSRLS). У production-runtime роли medcrm_app — нет BYPASSRLS.
  • Read replicas / BI: отдельная роль medcrm_analytics_ro с RLS включён, либо доступ только к денормализованным dw_fact_*.

5.5. CI-проверка RLS coverage

Тест проходит по information_schema.tables и проверяет, что у каждой таблицы с колонкой tenant_id включена RLS и есть хотя бы одна политика: 

SELECT t.table_name
FROM information_schema.tables t
JOIN information_schema.columns c USING (table_name, table_schema)
LEFT JOIN pg_class p ON p.relname = t.table_name
WHERE c.column_name = 'tenant_id'
  AND t.table_schema = 'public'
  AND p.relrowsecurity = false;
-- expect: 0 rows
Связь с ТЗ №1 Этот тест добавляется в CI prisma-job как обязательный шаг.

6. 2FA / MFA / WebAuthn

6.1. Политика 2FA

Роль Политика
Tenant admin Обязательно; нельзя отключить.
Главврач / Финансовый Обязательно.
Доктор Рекомендуется; tenant-admin может включить обязательность.
Регистратор / Кассир Опционально; tenant-admin может включить.
Пациент (portal) Опционально; включается из портала (ТЗ №5).

6.2. Эндпоинты auth-service

Метод Путь Описание
POST /auth/2fa/setup Генерирует TOTP secret, возвращает QR-код (otpauth://) и временный токен.
POST /auth/2fa/activate Принимает code; шифрует secret в БД (secretHash); генерирует 10 backup-codes (argon2-hashed).
POST /auth/2fa/verify Шаг challenge после login: принимает code, выдаёт final access + refresh.
POST /auth/2fa/disable Только при подтверждении паролем + code; запрещено для ролей с обязательным 2FA.
POST /auth/2fa/backup-codes/regenerate Инвалидирует старые, генерирует новые.
POST /auth/webauthn/register/begin (enterprise) Начало WebAuthn-регистрации.
POST /auth/webauthn/register/complete Завершение, сохранение public key.
POST /auth/webauthn/assert Аутентификация через WebAuthn (Touch ID, YubiKey).

6.3. Login flow

1. POST /auth/login { tenantCode, email, password }
   → 200 { status: "mfa_required", mfaToken } if user.2fa.isEnabled
   → 200 { accessToken, refreshToken } otherwise

2. POST /auth/2fa/verify { mfaToken, code }
   → 200 { accessToken, refreshToken, mfaSatisfiedAt }

3. /auth/bootstrap returns { user, tenant, modules, mfa: { required: bool, satisfied: bool } }

6.4. Frontend

  • Экран setup в Settings → Security: показывает QR, поле для кода, backup-codes (одноразовый просмотр + скачать PDF).
  • Step-up MFA challenge при логине — отдельный экран после пароля.
  • Recovery flow: «утерян аутентификатор» → backup code → если не помнит → tenant-admin может сбросить (с audit log).
  • Onboarding-баннер для tenant-admin, пока 2FA не включена.

6.5. Crypto-детали

  • TOTP secret: 160 бит, base32, окно ±1 (30s × 3).
  • Secret хранится как argon2id-hash? Нет — TOTP secret нужен для верификации, поэтому хранится зашифрованным симметрично (см. шифрование PHI): AES-256-GCM с key из KMS, не argon.
  • Backup-codes: 8-значные numeric, хранятся как argon2id-hash; одноразовые.
  • WebAuthn: @simplewebauthn/server.

7. Сессии и токены

  • Access token: JWT 15 минут, signed RS256 (асимметричный) — не HS256 — чтобы gateway мог проверять без shared secret.
  • Refresh token: opaque ID в Redis с rotation; срок 30 дней; одноразовый (rotation on use).
  • Каждой UserSession привязана пара deviceFingerprint (UA + IP-net + accept-language hash).
  • Endpoint GET /auth/sessions — список активных сессий пользователя.
  • Endpoint DELETE /auth/sessions/:id — revoke конкретной сессии.
  • Endpoint POST /auth/sessions/revoke-all — sign out everywhere (audit log).
  • Risk-based step-up: смена IP-сети + повышенно-важное действие (например, удаление пациента) → требовать 2FA-challenge даже внутри активной сессии.
  • Cookie strategy для frontend: HttpOnly, Secure, SameSite=Lax (для портала пациента — Strict).

8. Audit log + hash chain

8.1. Усиление модели

Добавляем поля в AuditLog:

model AuditLog {
  // ...existing
  prevHash      String?  @map("prev_hash") @db.VarChar(128)
  hash          String   @map("hash") @db.VarChar(128)
  signature     String?  @map("signature")    // optional KMS signature
}

hash = sha256(prevHash || canonical(record)) — каждая запись цепочно ссылается на предыдущую. Любое изменение прошлой записи ломает цепочку для всех последующих.

8.2. Append-only защита

  • Триггер Postgres BEFORE UPDATE OR DELETE на audit_logs бросает исключение для роли medcrm_app.
  • У роли medcrm_app отзываем UPDATE/DELETE GRANTs на audit_logs.
  • Раз в час BullMQ-задача (см. ТЗ №4) считает hash последних N записей и публикует чекпойнт в external storage (S3 / git) — для forensic.

8.3. Что писать в audit

Категория Действия
Auth login_success, login_failed, logout, 2fa_enabled, 2fa_disabled, password_reset, session_revoked
RBAC role_assigned, role_revoked, permission_granted, permission_revoked
Пациент patient_created, patient_viewed (PHI access), patient_updated, patient_deleted, patient_exported
EMR encounter_created, encounter_updated, encounter_signed, encounter_unsigned, prescription_created
Finance invoice_paid, invoice_refunded, cashier_shift_opened, cashier_shift_closed, refund_approved
System settings_changed, integration_key_rotated, module_enabled, module_disabled

8.4. PHI access logging

Просмотр карточки пациента — это событие. Добавляем декоратор @AuditPhiAccess('Patient') в Nest-контроллер, который записывает action: 'patient_viewed' с entityId. Это требование ФЗ-152 и аналогов.

9. Шифрование PHI (column-level)

9.1. Стратегия

  • Симметричный AES-256-GCM, ключи в KMS (AWS KMS / GCP KMS / HashiCorp Vault Transit / SOPS-encrypted с rotation).
  • Per-tenant DEK (Data Encryption Key), зашифрован master-KEK; cache в памяти процесса.
  • Шифруется на уровне приложения (Prisma extension), не на уровне БД. Это даёт изоляцию от Postgres-админа.
  • Поле в БД хранит JSON { "v": 1, "ct": base64, "iv": base64, "tag": base64 }.
  • Для search по зашифрованным полям — отдельный blind index (HMAC-SHA256 от lower-case значения).

9.2. Поля под шифрование

Таблица.поле Тип PHI Search
patients.first_name имя blind index по lower(firstName)
patients.last_name имя blind index
patients.phone контакт blind index (normalized e164)
patients.passport_number документ blind index (full match only)
patients.dob дата рождения не шифруем (для возрастных диапазонов)
medical_records.notes клинические записи не searchable
encounter_diagnoses.icd_text диагноз blind index по icd_code (не шифруем code, только text)
prescriptions.notes назначение не searchable

9.3. Миграция данных

Двухфазная:

  1. Добавить новые колонки *_enc и *_bi (blind index). Доуложить миграционный воркер, который перекладывает значения batch'ами по 1000 строк.
  2. После 100% перекладки — переключить read-path на *_enc; удалить старые колонки в отдельной миграции.

9.4. Ротация ключей

  • Master-KEK — ротация раз в 90 дней.
  • Per-tenant DEK — ротация раз в 180 дней или после инцидента.
  • Версионирование: запись содержит v и keyId; декрипт прозрачно поддерживает несколько активных ключей.
  • Скрипт ротации: создать новый DEK → перекодировать batch'ами → пометить старый как «retired».

10. Управление секретами

Перенос секретов из .env в централизованное хранилище:

Среда Хранилище Доступ
Локальная разработка SOPS + age-key (в репо лежит зашифрованный .env.dev.sops) Каждый разработчик имеет личный age-key, добавляется в SOPS recipients.
CI GitHub Actions Secrets Через secrets.* в workflow.
Staging / Production Doppler или HashiCorp Vault Каждый pod через service account; короткоживущие токены.

В коде запретить чтение process.env напрямую — только через типизированный ConfigService с Zod-валидацией.

  • Все secrets с TTL: 90 дней; ротация по календарю.
  • change_me_access_secret и change_me_refresh_secret в .env.example сохранить как «обязательно переопределить» — добавить отказ при старте, если значение в проде равно дефолту.
  • Совместно с ТЗ №1 — gitleaks-action блокирует случайный commit реальных секретов.

11. OWASP baseline и security headers

11.1. Headers (через Helmet config)

helmet({
  contentSecurityPolicy: {
    useDefaults: true,
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "'wasm-unsafe-eval'"],
      styleSrc: ["'self'", "'unsafe-inline'"], // until shadcn-only
      imgSrc: ["'self'", "data:", "blob:", "https://cdn.medcrm.app"],
      connectSrc: ["'self'", "wss:", "https://api.medcrm.app"],
      fontSrc: ["'self'", "data:"],
      objectSrc: ["'none'"],
      frameAncestors: ["'none'"],
      upgradeInsecureRequests: [],
    },
  },
  hsts: { maxAge: 63072000, includeSubDomains: true, preload: true },
  referrerPolicy: { policy: "strict-origin-when-cross-origin" },
  crossOriginOpenerPolicy: { policy: "same-origin" },
  crossOriginResourcePolicy: { policy: "same-site" },
  frameguard: { action: "deny" },
});

11.2. CORS

  • Whitelist по tenant (custom domain): https://<tenant>.medcrm.app и https://<custom>.
  • Credentials: true только для frontend-route группы; portal — отдельная policy.

11.3. Rate limiting (Redis-backed)

Текущий in-memory ограничитель в gateway заменяется на Redis-backed token bucket (либо @upstash/ratelimit):

Endpoint Limit
/auth/login 5 / 1 минута / IP + tenantCode
/auth/2fa/verify 5 / 1 минута / mfaToken
/auth/password/forgot 3 / 1 час / email
/auth/webauthn/* 10 / 1 минута / userId
Public API (/api/v1/*) 300 / минута / user (default)
Internal (/internal/v1/*) 1000 / минута / service

11.4. Input validation

  • Все DTO через Zod (уже стоит) + class-validator где исторически.
  • Маска полей в логах: phone, password, JWT (regex-redaction).
  • Strict content-type на POST/PUT; reject multipart/form-data на эндпоинтах, где он не ожидается.

11.5. File upload (MinIO)

  • Подписанные URL с TTL 15 минут.
  • Whitelist MIME-типов (application/pdf, image/png, image/jpeg, application/dicom).
  • Max size 50 МБ для документов; 200 МБ для DICOM.
  • Анти-вирус через ClamAV-sidecar (опционально) или Cloud Storage AV-API.

11.6. OWASP ZAP baseline

  • В CI weekly cron-job: zaproxy/action-baseline@v0.10 на staging.
  • Отчёт публикуется как artifact + комментарий в Security tab.
  • Threshold: 0 high, ≤ 5 medium на staging.

12. ФЗ-152 / Закон РТ о ПДн / GDPR

MedCRM работает в РФ, Таджикистане, потенциально Узбекистане. Требования у законов схожие, выровним по самому строгому (GDPR).

12.1. Subject Rights

Право Реализация
Доступ к данным Пациент в портале (ТЗ №5) запрашивает экспорт; backend генерирует FHIR Bundle + CSV (см. fhir-export) + PDF выписку.
Право на исправление Через портал и регистратора; меняется через стандартный flow + audit.
Право на удаление Tombstone-pattern: запись помечается deleted_at, PHI зануляется, audit сохраняется. Полное удаление через 7 лет (минимальный срок медицинской карты в РФ).
Право на ограничение Через tenant-admin → user request → запрет коммуникаций (CommunicationPreference).
Согласие Модель LegalDocumentTemplate + подпись (DigitalSignature). При first-visit пациент подписывает согласие.

12.2. Documentation

Создать набор обязательных документов:

  • DPA (Data Processing Agreement) с клиникой — клиника = controller, MedCRM = processor.
  • DPIA (Data Protection Impact Assessment) — на уровне продукта.
  • Privacy Policy для портала пациента.
  • Cookie Policy.
  • Notice-of-breach procedure (≤ 72 часа).
  • Регламент доступа к продакшен-БД (named users, MFA, session recording).
  • Регламент бэкапов и восстановления.
  • Регламент удаления данных (retention policy).

12.3. Локализация хранения

  • Tenant конфигурирует регион хранения: RU, TJ, UZ, EU.
  • Для каждого региона — изолированный кластер БД + S3 bucket.
  • Cross-region access запрещён (RLS + сетевые правила).
  • Этот шаг — preparation: реальное multi-region — отдельное ТЗ enterprise.

16. Acceptance criteria

  • На всех таблицах с tenant_id включена RLS; CI-проверка падает при нарушении.
  • Production app-роль medcrm_app не имеет BYPASSRLS; миграционная роль — имеет.
  • 2FA TOTP работает: setup → activate → verify; backup-codes одноразовые.
  • Tenant-admin и главврач не могут отключить свою 2FA.
  • WebAuthn регистрация и аутентификация работают (минимум Touch ID / YubiKey).
  • Refresh tokens в Redis, rotation on use, revoke per session работает.
  • JWT подписаны RS256; gateway проверяет через JWKS.
  • Rate-limit на /auth/login, /auth/2fa/verify, /auth/password/forgot работает консистентно в кластере.
  • audit_logs — append-only; UPDATE/DELETE falls для роли medcrm_app.
  • Hash-chain валидируется командой audit:verify; checkpoints публикуются раз в час.
  • Patient first/last name, phone, passport — зашифрованы; поиск работает через blind index.
  • Encryption keys в KMS; ротация per-tenant DEK выполнена хотя бы один раз.
  • CSP/HSTS/Permissions-Policy: оценка securityheaders.com ≥ A.
  • OWASP ZAP baseline зелёный (0 high, ≤ 5 medium).
  • Compliance docs (DPA template, Privacy Policy, Breach procedure) опубликованы.
  • Subject rights: тест-сценарий «пациент запросил экспорт + удаление» проходит end-to-end.

17.2. Риски

Риск Митигация
RLS включён, но Prisma middleware не выставил session var → запросы возвращают пусто. Health-check на старте: запрос-тест, который без session var должен возвращать 0 строк. Логи + alert.
Падение perf из-за SET в каждой транзакции. Use set_config(... , true) локально для транзакции; пул соединений через PgBouncer transaction-mode совместим.
Backfill PHI на больших таблицах вешает БД. Batched 1000 / row, throttle через BullMQ. Окно ночное; индексы создавать CONCURRENTLY.
Потеря KMS ключа = потеря всех данных. KMS multi-region replica; ежедневный backup encrypted-DEK в отдельный S3 с retention 365 дней; periodically run «recovery drill».
2FA включён → саппорт-нагрузка «забыл аутентификатор». Backup codes + recovery через tenant-admin (с audit). Документация для clinic-admin.
Hash-chain validator показывает разрыв → forensic-инцидент. Runbook: остановить запись, снять snapshot, исследовать source. Тренировка раз в квартал.
Юридическая трактовка «удаления» в РФ — нельзя удалить медкарту 7 лет. Tombstone-pattern с zero-out PHI и сохранением audit. Чёткое описание в Privacy Policy.
Расходы на KMS / Vault. AWS KMS ~$1/CMK/мес + $0.03 / 10k запросов — дешевле инцидента. На локальной разработке SOPS бесплатно.
Definition of Done RLS включена, 2FA обязательна для админов и работает, sessions revoke-aware, audit append-only с hash-chain, PHI шифруется на уровне колонок, secrets в KMS, OWASP-baseline зелёный, compliance pack опубликован. MedCRM может пройти аудит безопасности клиники-enterprise.
