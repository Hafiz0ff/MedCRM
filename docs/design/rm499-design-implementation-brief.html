ТЗ на перенос дизайна из прототипа rm-499 в MedCRM

Документ описывает, как применить визуальный и UX-подход из прототипа rm-499.created.app к текущему проекту MedCRM. Цель - привести операционный контур клиники к единой, плотной, спокойной SaaS-системе с левым меню, верхней панелью, компактными таблицами, статусами и рабочими дашбордами.

Источник: rm-499.created.app Дата анализа: 25 мая 2026 Target: MedCRM frontend Формат: HTML ТЗ

1. Объем анализа прототипа

В прототипе обнаружена единая shell-структура и 7 пунктов левого меню. Полностью оформлены 5 рабочих экранов: операционная, живая очередь, расписание, пациенты, врачи. Разделы «Отчёты» и «Настройки» сейчас являются заглушками с текстом «Отчёты в разработке», поэтому для них в этом ТЗ фиксируется будущая структура, но не буквальный перенос заглушки.

Раздел прототипа Состояние в rm-499 Целевой маршрут MedCRM Приоритет переноса
Операционная Полный dashboard с KPI, приемами, врачами и очередью /dashboard Высокий
Живая очередь Полная kanban-доска очереди /reception или отдельный режим /queue Высокий
Расписание Плотная таблица расписания дня /schedule Высокий
Пациенты Полный реестр пациентов с фильтрами и балансом /patients Высокий
Врачи Карточки загрузки врачей Новый раздел или блок в /dashboard Средний
Отчёты Заглушка /reports в будущем Низкий
Настройки Заглушка, пункт меню есть /settings Средний
Важно: прототип содержит визуальные кнопки «Новый пациент», «Записать», «Добавить», но в текущей сборке они не открывают отдельные окна. В MedCRM эти действия должны быть реализованы как drawer/modal states, а не оставаться декоративными кнопками.

2. Визуальная система, которую нужно перенести

Общий характер Светлый медицинский SaaS, низкий визуальный шум, много воздуха между секциями, но высокая плотность данных внутри таблиц и карточек.
Композиция Постоянный левый sidebar, фиксированная topbar, широкая рабочая область с контентом отступом 28-32px от краев.
Акцент Основной teal/emerald: логотип, active nav, primary buttons, положительные суммы, progress bars.
Статусы Зеленый - завершено/активно, синий - в кабинете/текущий прием, оранжевый - ожидает, красный - долг/риск/no-show, серый - архив/запланировано.

Цветовые токены

Primary #079685 / #009B8A Primary buttons, active nav, avatars, progress
Background #F8FAFC Основная рабочая область

Типографика

  • Основной шрифт: Inter.
  • Page title: 22-24px, weight 500-600. Не делать hero-заголовки.
  • Table header: 11-12px, uppercase, letter spacing умеренный, цвет muted.
  • Body/table text: 14px, имена пациентов и врачей weight 600.
  • Meta text: 12-13px, цвет muted.
  • KPI numbers: 26-30px, weight 500-600, без чрезмерного bold.

3. Глобальный App Shell

Визуальную основу прототипа нужно перенести на весь защищенный контур MedCRM.

Sidebar

  • Ширина desktop: 224px.
  • Фон: белый, правый border 1px solid #E4E7EC.
  • Верх: логотип-иконка в teal-квадрате 32px, название клиники «МедиЦентр» или актуальное название tenant, подпись «Частная клиника».
  • Навигация: иконка + label, высота пункта 42-44px, радиус 8px, active background #E7F8F5, active text #079685, left accent 2px.
  • Низ: пункт «Настройки», блок пользователя с аватаром initials, ролью и email, иконка выхода.
  • Collapsed state: по кнопке burger sidebar должен сжиматься до 72px; labels скрываются, tooltips обязательны.

Topbar

  • Высота: 58px.
  • Слева: burger, vertical divider, текущая дата «Пн, 25 мая 2026».
  • Справа: глобальный поиск 176-220px, refresh icon, notification bell с red dot, avatar.
  • Topbar фиксирована по высоте и не должна прыгать при загрузке данных.
  • Global search должен быть настоящим action: при фокусе открывать command/search popover с пациентами, записями и быстрыми действиями.

4. Экран: Операционная

Целевой route: /dashboard

Экран должен стать стартовым операционным пультом клиники. В прототипе он удачно совмещает KPI, приемы, загрузку врачей и очередь в одном viewport.

Пункт меню: Операционная Primary role: Администратор Desktop first

Структура экрана

  1. KPI strip на всю ширину: 6 карточек в одной строке.
  2. Основная колонка 60-65% ширины: список «Приёмы» за сегодня.
  3. Правая колонка 35-40%: «Врачи сегодня» и «Очередь сейчас».
  4. Никаких больших hero-секций; заголовки компактные.

KPI карточки

KPI Данные Действие по клику
Пациентов сегодня Количество записей/пациентов за день, delta к вчера. Открыть /schedule с текущей датой.
Завершено приёмов Формат 3 / 12, процент выполнения. Фильтр завершенных приемов.
В очереди Текущее число и среднее ожидание. Открыть queue board.
Выручка Сумма за день, delta к вчера. Финансовый отчет/касса, если доступно.
Не явились Количество и процент от записей. Фильтр no-show.
Загрузка Процент загрузки кабинетов, например «3 из 4 кабинетов». Открыть utilization panel.

Список приемов

  • Визуальный стиль: table/list hybrid, строки высотой 64-68px.
  • Слева фиксированное время моноширинным/табличным стилем.
  • Цветная vertical line у времени отражает статус: green completed, blue current, orange waiting, gray scheduled.
  • Пациент жирным, тип приема вторичной строкой.
  • Справа: врач, кабинет, статус badge.
  • Текущий прием получает бейдж «СЕЙЧАС» рядом с именем.

Правая колонка

  • «Врачи сегодня»: карточки врачей с initials avatar, специализацией, прогрессом 3/8, статусом «В кабинете» / «Перерыв».
  • Progress bar teal, ширина отражает процент принятых пациентов.
  • «Очередь сейчас»: компактные счетчики «Ожидают», «В кабинете», «Оформление», «Завершено».
  • Кнопка «Подробнее» ведет в queue board.

5. Экран: Живая очередь

Целевой route: /reception или /queue

В прототипе это отдельная kanban-доска для живого потока пациентов. В текущем MedCRM есть reception board; нужно привести его визуально и структурно к этому виду.

Структура

  • Header: eyebrow «ЖИВАЯ ОЧЕРЕДЬ», заголовок «Обновлено 10:47», live-dot «в эфире», справа primary «+ Добавить».
  • Board из 4 колонок: «Ожидает», «В кабинете», «Оформление», «Завершено».
  • Колонки одинаковой ширины, с внутренним counter badge справа в заголовке.
  • Фон страницы светло-серый; колонки белые с border и radius 10-12px.

Колонки и карточки

Колонка Карточка Статусный цвет
Ожидает ФИО, талон, врач, время ожидания. Один пациент может иметь red warning icon. Orange dot/counter.
В кабинете Выделенная active-card с blue border/background, label «НА ПРИЁМЕ», кабинет, врач, длительность. Blue.
Оформление ФИО, талон, действие: «Оформление рецепта», «Направление на МРТ», исполнитель. Teal.
Завершено ФИО, врач, время завершения, check icon. Green.

Обязательные действия

  • Кнопка «Добавить» открывает drawer быстрой постановки в очередь: пациент, причина, врач, приоритет.
  • Drag/drop между колонками меняет статус визита/очереди.
  • Клик по карточке открывает patient/visit preview drawer.
  • Время ожидания пересчитывается live и меняет цвет при превышении SLA.

6. Экран: Расписание

Целевой route: /schedule

Прототип использует не тяжелую календарную сетку, а плотную таблицу расписания дня. Для MedCRM нужно объединить это с уже реализованной логикой записи и click-to-book.

Header и фильтры

  • Eyebrow «РАСПИСАНИЕ».
  • Title: «Понедельник, 25 мая 2026».
  • Справа select «Все врачи» и primary «+ Записать».
  • Ниже optional segmented filter: Петров А.В., Кузнецова Е.И., Морозов С.Т.

Таблица расписания

  • Колонки: Время, Пациент, Тип, Врач / Кабинет, Статус.
  • Строки высотой 64-66px, много whitespace внутри row, но таблица остается плотной.
  • Время слева + vertical status marker.
  • Обеденный перерыв показывать отдельной disabled строкой: время 13:00, пациент «—», тип «Обеденный перерыв».
  • Статусы: «Завершён», «В кабинете», «Ожидает», «Запланирован».
  • Click-to-book: клик по свободному времени или кнопке «Записать» открывает drawer создания записи.

Drawer «Записать»

  • Поля: пациент, телефон/поиск, тип приема/услуга, врач, кабинет, дата, время, длительность, комментарий.
  • Если drawer открыт из строки/слота, дата, время и врач предзаполнены.
  • Показывать conflict warning, если врач/кабинет занят.
  • Primary action: «Создать запись». Secondary: «Создать и отметить приход».

7. Экран: Пациенты

Целевой route: /patients

Прототип задает хороший high-density CRM-реестр. Нужно перенести табличную структуру, фильтры и визуальные балансы.

Header и toolbar

  • Eyebrow «РЕЕСТР».
  • Title: «12 пациентов» или реальное количество из API.
  • Primary action справа: «+ Новый пациент».
  • Search input: «Поиск по имени, телефону, диагнозу...».
  • Filter buttons: «Все», «Активный», «Новый», «В архиве».
  • Secondary «Экспорт» с download icon.

Таблица

Колонка Что показывать
Пациент Avatar initials, ФИО, диагноз/заметка второй строкой.
Телефон Форматированный номер.
Д.Р. Дата рождения в формате dd.mm.yyyy.
Врач Ответственный врач или последний врач.
Визит Дата последнего или ближайшего визита.
Баланс Положительный teal, ноль gray, отрицательный red.
Статус Dot + label: активный, новый, в архиве.

Drawer «Новый пациент»

  • Кнопка должна открывать drawer справа.
  • Поля: ФИО или имя/фамилия, телефон, дата рождения, пол, ответственный врач, диагноз/заметка, филиал.
  • Inline duplicate check по телефону и ФИО.
  • После создания: toast + действия «Открыть карту» и «Записать на прием».

8. Экран: Врачи

Новый route: /doctors или часть personnel module

В прототипе раздел показывает 3 карточки врачей с текущей загрузкой. Это можно добавить как отдельный lightweight workspace или как drilldown из dashboard.

Карточка врача

  • Avatar initials 46px teal.
  • ФИО, специальность, статус справа: «В кабинете» / «Перерыв».
  • Три метрики: записей, принято, осталось.
  • Progress label + процент справа + progress bar.
  • Карточка кликабельна и открывает расписание врача на день.

Данные

  • Использовать существующие doctors/employees endpoints.
  • Расчет метрик брать из appointment status за выбранную дату.
  • Добавить состояние empty: «На выбранную дату врачей с расписанием нет».

9. Отчёты и Настройки

В прототипе оба раздела сейчас не проработаны: после клика отображается «Отчёты в разработке». В MedCRM нельзя переносить одинаковую заглушку для двух разных разделов. Нужно сделать осмысленные placeholder surfaces.

Отчёты

  • Route: /reports или текущий BI module.
  • Показать cards: операционный отчет, финансовый отчет, загрузка врачей, no-show, экспорт пациентов.
  • Каждая card: title, short description, status «скоро» или action «Открыть», если backend уже есть.

Настройки

  • Route: /settings.
  • Показать настройки клиники, филиалов, пользователей, ролей, модулей, интеграций.
  • Сохранить визуальный стиль прототипа: белые панели, тонкие borders, компактные строки.

12. Состояния и интеракции

  • Loading: skeleton на KPI, таблицы, queue columns, doctor cards.
  • Empty: сохранение layout, понятный CTA; не показывать огромные пустые области без смысла.
  • Error: компактная панель с retry action.
  • Permission-limited: скрывать primary actions, если нет прав.
  • Realtime update: мягкая подсветка измененной строки/карточки.
  • Hover: строки таблиц и карточки должны иметь subtle background.
  • Selected: выбранная карточка или строка получает teal/blue outline.
  • Drag-over: queue columns подсвечиваются border и background.

13. Responsive требования

  • 1440x900: основной reference viewport, должен выглядеть как прототип.
  • 1366x768: таблицы и board должны помещаться без вертикального перегруза; допускается внутренний scroll.
  • 1024px: sidebar можно collapsed, dashboard right column уходит ниже.
  • 390px mobile: таблицы превращаются в card list, sidebar становится drawer, topbar search скрывается в icon.
  • Никакой текст не должен вылезать из кнопок, бейджей и таблиц на русском языке.

15. Критерии приемки

  • Визуальное соответствие: MedCRM должен выглядеть как тот же продуктовый язык, что rm-499: светлый canvas, teal accent, compact SaaS layout, плотные таблицы.
  • Shell: sidebar/topbar едины на всех защищенных страницах.
  • Операционная: есть 6 KPI, приемы, врачи, очередь в одном viewport.
  • Очередь: есть 4 колонки с карточками и live/waiting semantics.
  • Расписание: есть дневная таблица с врачом, кабинетом, длительностью, статусами и drawer записи.
  • Пациенты: есть high-density table, фильтры, баланс, диагноз/заметка, drawer нового пациента.
  • Врачи: есть карточки загрузки врачей с progress.
  • Стабильность: проходят npm --workspace frontend run typecheck, npm --workspace frontend run build, backend e2e не ломаются.
  • Не принимается: перенос только цветов без структуры, декоративные кнопки без действий, таблицы без loading/empty/error states, разные стили статусов между экранами.
