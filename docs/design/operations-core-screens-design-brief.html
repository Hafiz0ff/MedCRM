ТЗ на дизайн четырех операционных экранов MedCRM

Задача дизайнера: переработать UI для базового операционного контура клиники так, чтобы администратор, регистратор и оператор расписания могли быстро видеть состояние смены, находить пациента, управлять очередью и создавать запись без лишних переходов.

Проект: MedCRM Контур: Operations Core Формат результата: Figma Экраны: 4 desktop + responsive

1. Контекст и цель

Сейчас интерфейс функционально уже показывает основные страницы, но визуально выглядит как технический прототип: много пустого пространства, слабая иерархия, разрозненные карточки, недостаточно плотности данных для рабочего SaaS-инструмента, слабая поддержка критических состояний смены. Нужно сделать дизайн не маркетинговым и не декоративным, а операционным: спокойным, плотным, предсказуемым, удобным для повторяющейся работы в клинике.

Важно: дизайнеру нужно отрисовать не лендинг, а реальные рабочие поверхности. Первый экран каждой страницы должен сразу быть инструментом: таблица, очередь, календарь, KPI, фильтры, быстрые действия.

Не входит в это ТЗ

  • Полный EMR/EHR workspace, медицинская карта, назначения, подпись приема.
  • Финансы, склад, BI, коммуникации, настройки.
  • Редизайн бизнес-логики или API. Дизайн должен учитывать текущие возможности, но может предложить более правильное расположение элементов.

2. Главная дизайн-задача

Сделать интерфейс уровня production SaaS для клиники. Он должен выглядеть как инструмент, которым пользуются весь день: понятный с первого взгляда, быстрый, спокойный, с высокой плотностью полезной информации и четкими действиями.

Визуальная гипотеза Светлая медицинская операционная система: чистая поверхность, темная постоянная навигация, один основной teal-акцент, аккуратные статусы, компактные таблицы, понятные зоны действия.
UX-гипотеза Пользователь должен понимать состояние смены за 5 секунд, находить пациента за 10 секунд, создавать запись за 30-60 секунд, менять статус визита в 1-2 действия.
Интеракционная гипотеза Быстрые действия должны быть ближе к объекту действия: у пациента, визита, слота, очереди. Большие переходы оставляем только для полной карточки.
Ограничение Не делать интерфейс «красивой CRM-картинкой». Нужна утилитарная рабочая среда, где статус, время, пациент, врач и действие читаются быстрее декоративных элементов.

3. Общие требования ко всем четырем экранам

3.1. App Shell

  • Сохранить концепцию: темный левый сайдбар + светлая рабочая область.
  • Сайдбар должен быть компактным, но читаемым: логотип, клиника, группы меню, активный пункт, заблокированные/скоро-модули, статус активного модуля.
  • Активный пункт меню должен быть очевиден без сильного визуального шума.
  • Верхняя панель должна включать: текущая клиника/филиал, глобальный поиск, live-индикатор, уведомления, профиль.
  • Глобальный поиск должен визуально быть системным элементом, а не локальным полем страницы. Placeholder: «Найти пациента, запись или действие».
  • Высота верхней панели должна быть стабильной. Никакие состояния не должны прыгать при загрузке.

3.2. Layout и плотность

  • Основной desktop target: 1440 x 900. Также проверить 1366 x 768, 1280 x 800, 1024 x 768.
  • Рабочая область должна использовать ширину рационально. Не оставлять большие пустые зоны, если есть полезные данные или действия.
  • Для операционных экранов использовать максимум два уровня контейнеров. Не вкладывать карточки в карточки без необходимости.
  • Карточки использовать только для самостоятельных объектов: KPI, визит, пациент, слот, alert, форма. Секции лучше оформлять как панели/области.
  • Все таблицы и board-колонки должны иметь фиксированную структуру, чтобы пустое/загруженное состояние не меняло общую сетку.

3.3. Цвет и статусы

  • Основной акцент: teal/green для primary action и live/ready состояний.
  • Информационные статусы: blue. Предупреждения: amber. Ошибки/отмена: red. Нейтральные: slate/gray.
  • Статусы должны иметь единую систему бейджей на всех экранах: размер, фон, текст, иконка при необходимости.
  • Цвет не должен быть единственным способом различения статуса. Добавить текст, иконку или форму бейджа.

3.4. Typography

  • Типографика должна быть компактной: page title крупный, но без hero-ощущения; таблицы и панели плотные.
  • Не использовать огромные заголовки внутри рабочих панелей.
  • Лейблы, числа KPI, имя пациента, время визита и статус должны иметь четкую иерархию.
  • Нельзя использовать отрицательный letter-spacing. Текст должен помещаться в кнопки и бейджи на русском.

3.5. Состояния

Для каждого экрана обязательно отрисовать следующие состояния:

  • Loaded with data: нормальное заполненное состояние.
  • Empty: нет записей/нет пациентов/нет очереди, но layout остается стабильным.
  • Loading: skeleton для таблицы, KPI, board-колонок, правого сайдбара.
  • Error: недоступен backend, нет доступа, ошибка сохранения.
  • Permission-limited: действие скрыто или disabled, если у роли нет права.
  • Realtime update: мягкая подсветка обновленного визита/строки/счетчика.

3.6. Responsive

  • Desktop: основной рабочий режим.
  • Tablet 1024: сайдбар может стать уже, правые панели уходят ниже или в drawer.
  • Mobile 390: дизайн нужен как адаптивная спецификация, даже если основной сценарий desktop. Таблицы превращаются в списки, board получает горизонтальный scroll, формы становятся bottom sheet / full screen.

4. Компонентная система

Дизайнер должен собрать минимальную дизайн-систему для этих экранов, чтобы разработчик мог переносить UI без разночтений.

Обязательные компоненты

  • App shell: sidebar, topbar, global search, branch selector/state, user menu, notification button.
  • Buttons: primary, secondary, ghost, icon-only, destructive, disabled, loading.
  • Inputs: text input, search input, select, date picker, time picker, textarea, phone input.
  • Filters: filter button, active filter chips, filter popover/panel, reset action.
  • Badges: appointment status, patient status, priority, live, payment, conflict.
  • KPI cards: normal, warning, critical, loading.
  • Data table: compact row, selected row, hover, empty, loading, pagination/virtualization hint.
  • Patient identity: avatar/initials, full name, patient code, contact line, tags.
  • Appointment card: time, patient, service, doctor, cabinet, status, quick actions.
  • Kanban/board column: header counter, empty, drag-over, overflow.
  • Calendar slot: empty, booked, conflict, outside working hours, blocked, selected.
  • Right panel: creation form, context summary, queue, utilization.
  • Slide-over / drawer: patient preview, appointment preview, filters, create appointment.
  • Toast / confirmation dialog: success, error, warning, cancel confirmation.

Figma tokens

  • Colors: background, surface, nav, border, muted text, primary, primary-hover, status colors.
  • Typography: page title, section title, table header, table body, meta text, badge text, button text.
  • Spacing: 4/8/12/16/20/24/32 scale.
  • Radii: small 6px, panel 8px, pill 999px. Не использовать большие «пузырьковые» радиусы для рабочих поверхностей.
  • Elevation: минимальные тени только для floating/drawer/dropdown; основные панели через border и фон.

5. Экран 1: Операционная панель

/dashboard - краткий контроль смены

Это стартовый экран после входа. Его задача - дать управляющему или администратору состояние дня: сколько записей, кто уже в клинике, есть ли конфликты, насколько загружены врачи, какие действия требуют внимания.

Image #3 Role: admin / shift manager Priority: high

Что сейчас не хватает

  • Слишком мало операционной информации для dashboard.
  • KPI выглядят как статичные карточки, но не помогают быстро перейти к проблеме.
  • Блок «Состояние системы» занимает много места и менее важен для ежедневной работы, чем риски смены.
  • Нет выраженного слоя alert/action queue: опоздания, конфликты, ожидание оплаты, не подтвержденные записи.

Новая структура экрана

  1. Header: заголовок «Операционная панель», дата/филиал, primary action «Новая запись», secondary «Пациенты».
  2. KPI strip: 4-6 компактных показателей в одну строку на desktop.
  3. Operational alerts: список задач смены, сгруппированный по срочности.
  4. Today flow: мини-воронка статусов визитов: запланировано, пришли, на приеме, к оплате, завершено, отмены.
  5. Doctor/cabinet load: компактный список загрузки врачей или кабинетов с progress bar.
  6. System readiness: сделать компактным secondary-блоком, не главным контентом.

KPI требования

  • «Записи сегодня» - число, подтвержденные, требуют звонка.
  • «Пациенты в клинике» - число, ожидают администратора, ожидают врача.
  • «Загрузка врачей» - процент, peak time.
  • «Очередь» - сколько ожидают и максимальное время ожидания.
  • «К оплате» - количество завершенных приемов без оплаты.
  • «Риски» - конфликты ресурсов, опоздания, неподтвержденные записи.

Интеракции

  • Клик по KPI ведет в соответствующий фильтр на `/reception` или `/schedule`.
  • Alert row имеет primary next action: «Проверить», «Позвонить», «Открыть запись», «Назначить кабинет».
  • Realtime обновление должно подсвечивать изменившийся KPI/alert на 1-2 секунды.

6. Экран 2: Регистратура

/reception - live board потока пациентов

Главный рабочий экран регистратора. Он должен показывать поток пациентов по статусам и давать быстрые действия без открытия полной карточки.

Image #1 Role: receptionist Realtime board

Что сейчас не хватает

  • При пустых данных почти весь экран занят пустыми колонками, нет ощущения полезного инструмента.
  • Канбан-колонки слишком растянуты и малоинформативны.
  • Правая очередь визуально оторвана от board и не показывает next actions.
  • Нет хорошо продуманной карточки визита: время, пациент, врач, услуга, кабинет, статус, быстрые действия.

Новая структура экрана

  1. Header: «Регистратура», live indicator, поиск пациента, «Быстрая запись».
  2. Counter strip: 6 счетчиков статусов, компактнее текущего варианта.
  3. Main board: горизонтальные колонки статусов с фиксированной шириной и drag/drop.
  4. Right context rail: очередь, последние check-in, звонки/быстрая связь, selected patient preview.
  5. Bottom/secondary area: если данных мало, показать ближайшие записи и подсказки быстрых действий, а не большое пустое поле.

Колонки board

Колонка Смысл Быстрые действия
План Запланированные и подтвержденные визиты. Отметить приход, позвонить, отменить.
Ожидают Пациенты уже в клинике. Начать прием, повысить приоритет, открыть пациента.
На приеме Активные приемы у врача. Завершить, перейти к EMR, открыть запись.
К оплате Прием завершен, нужна оплата. Перейти к оплате, отправить счет.
Завершено Завершенные визиты. Открыть детали, повторная запись.
Отмены Отмененные и no-show. Перезаписать, отметить причину.

Карточка визита

  • Верхняя строка: время и статус.
  • Основная строка: ФИО пациента + patient code.
  • Meta: услуга, врач, кабинет, источник записи.
  • Risk flags: опоздание, конфликт, VIP, долг, нет контакта, первая запись.
  • Actions: иконки или компактные кнопки. Основное действие видно, вторичные - в меню.

Drag and drop

  • Drag-over состояние колонки должно быть очевидным.
  • При недопустимом переносе показать предупреждение и объяснение.
  • После переноса статус обновляется optimistic UI + toast.
  • Нужен дизайн подтверждения отмены визита.

7. Экран 3: Пациенты

/patients - единый реестр и быстрый пациент

Экран должен работать как CRM-реестр: быстро найти пациента, увидеть статус/контакт/филиал, открыть карточку, создать нового пациента с минимальными полями и заметить возможные дубли.

Image #2 Role: receptionist / admin Data table

Что сейчас не хватает

  • Таблица выглядит технически и не показывает достаточно пациентского контекста.
  • Форма «Новый пациент» слишком простая, не учитывает дубль, источник, согласия, быстрый переход к записи.
  • Фильтры спрятаны, но активные фильтры не представлены как рабочий слой.
  • Нужно больше сценариев: создание, дубликаты, пустой поиск, ошибка, нет прав.

Новая структура экрана

  1. Header: «Пациенты», count result, action «Создать пациента».
  2. Search/filter bar: большой поиск, фильтры, активные chips, сортировка.
  3. Patient table: основной список с плотными строками.
  4. Right rail: новый пациент или selected patient preview, в зависимости от контекста.
  5. Duplicate warning area: если найдены возможные дубли, показать отдельный компактный блок над таблицей.

Колонки таблицы

  • Пациент: avatar initials, ФИО, patient code, возраст/дата рождения при наличии.
  • Контакт: основной телефон, иконка канала, fallback «нет контакта».
  • Статус: новый, активный, спящий, VIP, ограничен.
  • Теги: 1-3 видимых тега + overflow.
  • Филиал: название филиала.
  • Последний визит / следующая запись: дата, если есть.
  • Действия: открыть, записать, меню.

Форма нового пациента

  • Минимальное состояние: имя, фамилия, телефон, филиал.
  • Расширенное состояние: дата рождения, пол, источник, комментарий, согласие на коммуникации.
  • При вводе телефона или ФИО показать inline duplicate check.
  • После создания предложить два действия: «Открыть карточку» и «Создать запись».
  • Ошибки валидации должны быть на уровне поля, не только toast.

Фильтры

  • Статус пациента.
  • Теги.
  • Филиал.
  • Есть долг / нет долга, если доступно.
  • Есть будущая запись / без записи.
  • Дата последнего визита.

8. Экран 4: Расписание

/schedule - календарь записи и загрузка кабинетов

Экран должен дать оператору быстрый способ выбрать день, увидеть расписание врачей/кабинетов, создать запись и отследить конфликты ресурсов.

Image #4 Role: scheduling operator Calendar

Что сейчас не хватает

  • При пустом дне большая центральная область почти бесполезна.
  • Нет плотного календарного grid для реального расписания.
  • Форма создания записи в правом rail слишком низко и может обрезаться.
  • Загрузка кабинетов отделена от календаря, хотя это ключевой фактор записи.
  • Нет визуального дизайна конфликтов: врач, кабинет, услуга, буфер, пересечение времени.

Новая структура экрана

  1. Header: «Расписание», actions «Фильтры», «Создать запись».
  2. Calendar toolbar: date picker, prev/next, today, day/week toggle, doctor/cabinet filters.
  3. Main calendar: day view с колонками врачей или кабинетов; week view как compact schedule.
  4. Right rail: selected slot/appointment details, create appointment form, room utilization.
  5. All appointments list: ниже календаря или drawer, не должен конкурировать с основным календарем.

Calendar states

  • Empty day: показать рабочую сетку времени, availability hints и CTA создать запись, а не пустой блок.
  • Booked slot: пациент, услуга, статус, длительность, цвет статуса.
  • Selected slot: правый rail раскрывает форму создания записи с выбранным временем.
  • Conflict slot: красно/amber outline, tooltip/popover с причиной.
  • Blocked time: серый фон, причина блокировки.
  • Outside working hours: muted background.
  • Drag appointment: ghost preview, target slot highlight.

Форма создания записи

  • Пациент: searchable select + быстрый create patient.
  • Услуга: select с длительностью и ценой, если есть.
  • Врач: select с availability state.
  • Дата и время: prefilled из выбранного слота, ручное редактирование.
  • Кабинет: auto-selected/recommended + conflict warning.
  • Комментарий, источник записи, reminder option.
  • Primary action: «Создать запись». Secondary: «Создать и отметить приход» для walk-in сценария.

Room utilization

  • Показать utilization bar по кабинетам.
  • Показать минуты занятости и свободные окна.
  • Показать warning при перегрузке или конфликте.
  • Клик по кабинету фильтрует календарь.

10. Accessibility и usability

  • Контраст текста и статусов должен соответствовать WCAG AA.
  • Все кликабельные элементы минимум 32px по высоте на desktop и 40px на touch.
  • Иконки в icon-only buttons должны иметь tooltip/label.
  • Фокус-состояния обязательны для keyboard navigation.
  • Таблицы должны иметь понятные hover/selected/focus states.
  • Не использовать только цвет для статусов.
  • Все empty/error states должны объяснять следующий шаг.

11. Требования к Figma deliverables

  1. Page 1: Foundations - colors, typography, spacing, status system.
  2. Page 2: Components - app shell, buttons, inputs, badges, tables, appointment cards, calendar slots, forms, drawers.
  3. Page 3: Desktop screens - четыре основных экрана в 1440 x 900.
  4. Page 4: Screen states - loading, empty, error, permission-limited, selected, drag-over, realtime update.
  5. Page 5: Responsive - 1024 tablet and 390 mobile key adaptations.
  6. Page 6: Prototype - основные переходы: dashboard to schedule, patients create, reception status transition, schedule create appointment.
  7. Annotations - размеры, sticky behavior, scroll behavior, state rules, компонентные variants.
Без состояний и component variants дизайн считается неполным. Нужны не только четыре красивые картинки, а рабочая спецификация для реализации.

12. Критерии приемки дизайна

  • Понятность: по первому экрану ясно, что происходит в клинике сейчас.
  • Скорость: основные действия видны без поиска: создать запись, найти пациента, отметить приход, открыть карточку.
  • Плотность: интерфейс использует desktop-пространство как рабочий инструмент, не как лендинг.
  • Единая система: статусы, кнопки, поля, таблицы и карточки выглядят одинаково на всех экранах.
  • Состояния: есть дизайн для loading, empty, error, no permission, realtime update, drag/drop.
  • Responsive: есть понятное поведение для tablet/mobile.
  • Не принимается: декоративные hero-блоки, огромные пустые карточки, непонятные цвета статусов, перегруженные тени/градиенты, дизайн только для одного идеального состояния.
